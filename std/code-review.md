# 代码审查标准

## 审查范围

0. 直接修改文件
1. 受影响调用方
2. 受影响被调用方
3. 数据结构
4. API 契约
5. 权限和安全边界
6. 错误处理
7. 幂等性
8. 事务边界
9. 缓存和 MQ 行为
10. 数据库表设计、索引、SQL、迁移和回滚方案
11. 配置链路
12. 测试文件
13. 文档

## 审查清单

### 功能正确性

0. 代码是否满足需求
1. 边界条件是否处理
2. 异常路径是否覆盖
3. 并发场景是否安全
4. 幂等性是否满足

### 代码质量

0. 是否遵循语言编码标准
1. 方法是否过长
2. 命名是否清晰
3. 注释是否必要且准确
4. 是否存在重复代码
5. 是否引入不必要的依赖

### 性能

0. 是否存在 N+1 查询
1. 是否存在循环中的 IO
2. 大数据量处理是否合理
3. 缓存使用是否得当

### 安全

0. 输入是否校验
1. 是否存在 SQL 注入、XSS 风险
2. 敏感信息是否泄露
3. 权限校验是否完整
4. 是否有越权风险

### 可维护性

0. 是否易于理解和修改
1. 是否包含足够测试
2. 文档是否同步更新
3. 是否存在技术债务

## 问题分级

0. 阻断：必须修复，否则不能合并
1. 高风险：建议本次修复
2. 中风险：可以后续修复，但需跟踪
3. 建议：供作者参考

## 审查态度

0. 对事不对人
1. 解释"为什么"，而不是简单否定
2. 提出具体修改建议
3. 认可好的设计和实现
4. 不追求个人代码风格偏好

## 审查后

0. 所有阻断问题修复后重新审查
1. 高风险问题修复后确认
2. 中风险和建议问题记录到 issue 或 TODO
3. 审查通过后批准合并
