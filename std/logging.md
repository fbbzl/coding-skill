# 日志与可观测性标准

## 日志

### 日志级别

0. DEBUG：详细的调试信息，生产环境通常关闭
1. INFO：正常业务流程记录
2. WARN：非预期但可以自动恢复的情况
3. ERROR：需要人工介入的错误
4. FATAL：系统级致命错误

### 日志内容

0. 使用结构化日志，包含时间、级别、trace_id、服务名
1. 入口方法记录请求参数（脱敏后）
2. 出口方法记录响应摘要
3. 异常路径记录完整堆栈和上下文
4. 关键业务节点记录状态变化
5. 禁止在循环中打印日志

### 日志规范

0. 使用占位符风格，避免字符串拼接
1. 日志消息使用中文或英文统一，不混用
2. 敏感信息脱敏：手机号、身份证、银行卡、密码、Token
3. 不要记录过大对象，记录关键字段即可
4. 异步/分布式场景必须包含 trace_id

## 指标

0. 暴露业务指标：QPS、成功率、延迟分位值
1. 暴露系统指标：CPU、内存、磁盘、连接数
2. 使用 Prometheus / Grafana 或等效方案
3. 指标命名规范：`service_module_metric_unit`
4. 标签维度合理，避免高基数

## 追踪

0. 分布式请求必须携带 trace_id
1. 跨服务调用传递 trace context
2. 关键 Span 标注操作类型和耗时
3. 采样率按环境配置，生产建议 1%~10%

## 告警

0. 告警必须包含现象、影响、建议操作
1. 避免告警疲劳，合并同类告警
2. 设置合理的阈值和恢复通知
3. 关键告警必须电话/短信通知
4. 告警必须有值班响应 SOP

## 日志存储

0. 结构化日志统一收集
1. 保留策略按级别区分：ERROR 至少 30 天，INFO 7~30 天
2. 支持关键字搜索和链路追踪
3. 审计日志单独存储，保留 180 天以上
4. 日志存储成本可控，定期归档
