# 测试标准

## 测试分层

0. 单元测试：验证函数/类行为，Mock 外部依赖
1. 集成测试：验证模块间交互，使用真实中间件
2. 端到端测试：验证完整业务流程
3. 契约测试：验证服务间 API 契约
4. 性能测试：验证吞吐量和延迟

## 测试原则

0. 测试代码是产品代码，必须保持可读和可维护
1. 每个测试只验证一个行为
2. 测试名必须清晰表达场景和期望
3. 使用 Arrange-Act-Assert 结构
4. 不要为通过测试而写测试

## 单元测试

0. 核心业务方法必须有单元测试覆盖
1. 分支覆盖率目标 80% 以上
2. 边界条件必须测试
3. 异常路径必须测试
4. Mock 只用于外部依赖，不 Mock 被测对象内部

## 集成测试

0. 数据库操作使用测试数据库或事务回滚
1. 消息队列测试使用内存版或测试专用队列
2. 外部 HTTP 调用使用 WireMock 或类似工具
3. 测试数据独立，不依赖已有数据

## 端到端测试

0. 覆盖核心用户旅程
1. 每个流程至少一个 happy path 和一个异常 path
2. 测试数据通过 API 或脚本准备
3. 运行环境尽可能接近生产

## 测试数据

0. 使用 Factory/Object Mother 模式创建测试数据
2. 不共享测试状态
3. 测试完成后清理数据

## 运行策略

0. 本地提交前必须跑通单元测试
1. CI 中必须跑通单元测试 + 集成测试
2.  nightly 跑端到端测试
3. 测试失败禁止合并

## 性能与稳定性

0. 新增性能敏感代码必须补充基准测试
1. 并发代码必须补充并发测试
2.  flaky 测试必须修复，不允许忽略
3. 测试执行时间过长的要优化
