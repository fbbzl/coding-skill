# 通用编码原则

本标准适用于所有语言和框架，是最基础的工程纪律。

## 核心三要素

0. 简单：如无必要，勿增实体
1. 合适：选择最匹配当前场景的方案，而不是最新或最炫的方案
2. 可扩展：为未来变化预留扩展点，但不要过度设计

## 优先级

```
可读性 > 正确性 > 可扩展性 > 性能
```

如果为性能牺牲了可读性，必须有明确的证据和审批。

## 不可触碰的红线

0. 不允许在循环中进行任何 TCP 通信 → 先批量聚合再请求
1. 不允许在循环中执行 SQL → 使用批量查询或 IN 语句
2. 不允许在循环中打印日志 → 循环结束后再统一输出摘要
3. 不允许在循环中读取大文件 → 使用流式处理或预加载
4. 不允许暴露密钥、密码、Token 到日志或前端 → 统一脱敏
5. 不允许使用明文存储密码 → 必须哈希加盐
6. 不允许信任用户输入，必须校验和转义 → 服务端校验是最后防线

## 造轮子原则

0. 造轮子之前，必须确认现有三方工具中无该实现
1. 如果业务逻辑是领域特定的，可以自建；如果是通用能力，优先复用
2. 自建的轮子必须有测试、文档和维护者

## 代码质量

0. 单个方法体不超过 40 行
1. 单个文件尽量控制在 400 行以内
2. 圈复杂度不超过 10，超标则拆分
3. 嵌套层级不超过 4 层
4. 参数数量不超过 5 个，超过用对象封装
5. 所有代码能对齐的尽量对齐

## 可读性

0. 使用有意义的命名，避免缩写和魔术数字
1. 注释解释"为什么"，而不是"做什么"
2. 复杂逻辑必须写伪代码或流程说明
3. 避免长链式调用和深层嵌套
4. 保持一致的代码风格

## 可维护性

0. 一个类/模块只做一件事
1. 依赖关系单向流动，禁止循环依赖
2. 公共 API 变更必须向后兼容或走废弃流程
3. 删除代码要彻底，不留僵尸代码
4. 重构必须伴随测试

## 交付纪律

0. 提交代码前必须本地自测通过
1. 提交信息必须说明做了什么和为什么
2. PR 描述必须包含变更范围和验证方式
3. 代码审查不通过禁止合并
4. 上线前必须通过 QA 验收
