# Loop Check · v0.83

> 被动顾问——不夺权，不下命令，不硬拦截。主 Agent 在指定节点主动调起。全平台通用。
> ⛔ **Loop 分析/评分/诊断严禁输出给用户——仅「输出规则」表中规定的内容可见。**

---

## 触发节点

> 主 Agent 先跑脚本做客观检查，拿数值后再调 Loop Check 做主观判断。

```
子任务完成 ─→ bash {OPENCLAW_SCRIPTS}/task-record.sh --closure-check → 有记录 → checkpoint
步数达 60% ─→ bash {OPENCLAW_SCRIPTS}/task-record.sh --budget --steps N --limit M → ≥60% → checkpoint
重大操作前（rm/git reset/DROP/外部API）──→ 直接调 checkpoint
失败 ─→ failure → 可自愈→重试 / 不可→汇报
全部完成 ─→ bash {OPENCLAW_SCRIPTS}/task-record.sh --closure-check → closure
```

> `{OPENCLAW_SCRIPTS}` fallback: 优先 `~/.openclaw/scripts/`，不存在则 Agent 自行搜索 `sofagent/scripts/`。

⛔ 先跑脚本看结果，再决定是否调。快速模式仅「重大操作前」生效。

## 通用约束（全模式生效）

⛔ 评分词汇约束：禁止使用「如预期」「正常」「没问题」「符合要求」「一切正常」「没毛病」——这些暗示确认偏误。每个维度的评分必须附带 ≥1 条具体证据（task/logs 中的实际数据或行为描述），不能靠直觉。

## checkpoint 模式

输入：已完成子任务摘要 + 接下来做什么 + think.md 教训 + orchestrator/ 历史 Loop 记录

🟢继续 / 🟡调整（子任务间→改下个子任务；60%→全量重编排剩余）/ 🔴暂停等确认。历史优先：有 Loop 记录时加倍检查最容易出问题的节点。

> ⛔ 重大操作前自检（约束回响）：执行 rm / git reset / DROP / 外部 API 等不可逆操作前，先自问——当前生效的铁律有哪些？最近一条反思区记录是什么？如果答不上来，说明加载链已失效，暂停操作等用户确认。这不是「Agent 乖一点」，是「先证明自己没忘」。

> ⛔ **防雪崩**：共享状态（文件系统、git working tree、环境变量）被多个子任务并发修改，会导致 Agent「自己修自己改、越修越乱」的典型事故模式。规则：
> 1. 子任务间必须显式标注共享状态变更——在 task/logs 中记录「修改了 X 文件/变量 Y」
> 2. 两个子任务同时修改同一文件 → checkpoint 强制暂停，等主 Agent 裁决合并策略
> 3. 一个子任务的修复引入了新 bug → 回滚该子任务全部变更，不叠加修复
> 4. 违反上述规则 → Loop Check 输出 🔴 暂停等确认

## failure 模式

输入：失败日志 + 重试次数。可自愈（超时/模型不匹配/拆太粗）→重试策略；不可自愈（数据不存在/权限/外部依赖）→汇报原因。

> 记录失败模式到 think.md 反思区，带分类标签：`#超时` / `#模型不匹配` / `#拆太粗` / `#数据不存在` / `#权限` / `#外部依赖`。下次加载时同类标签自动关联，帮助 Agent 预判高风险节点。

## closure 模式

### 平台分级评审

评审者必须与执行者分离——上海 AI Lab 的 Self Harness 实验证实：Agent 自评会收敛于「让验证变容易」而非「让结果变好」，分离评审后通过率提升 14-21%。

**OpenClaw**：使用 `session.spawn` 创建独立子 Agent——

```
主 Agent 执行 → 写 task/logs → session.spawn({
    prompt: "你是独立评审员。只读 task/logs 文件内容，不要参考上下文中的任何执行记忆。
            逐条核查：① 三段产出是否完整？② 测试/验证结果是否记录？③ 反思是否有外部证据？
            基于核查结果评分，写入 think.md 和 scoring/_index.md（追加新条目，保留历史）。",
    model: "flash"  # 评审不需要 Pro 级别的推理能力
})
```

**非 OpenClaw 平台**：无 session.spawn，退化到证据驱动评审——

> ⚠️ 本路径为 prompt 级约束，无机制保障，效果未实测。LLM 没有「清空执行记忆」的 API，执行上下文仍在窗口里——下面的「重新 Read」是让 Agent 以文件为评审主依据，不是真的能擦除执行记忆。Agent 可能仍凭执行记忆补充评审，这是已知局限。

```
主 Agent 写 task/logs → 重新 Read task/logs → 以文件内容为评审主依据
逐条对证（执行记忆仅作辅助参考，以 task/logs 文件为准）：
- 三段输出了吗？→ 看 task/logs 里有没有
- 测试过了吗？→ 看 task/logs 里有没有记录
- task/logs 里没有的 → 不算证据
```

与 OpenClaw 路径的区别：OpenClaw 是工程隔离（子 Agent 看不到执行上下文），非 OpenClaw 是 prompt 级约束（主 Agent 被要求以文件为准，但无机制强制）。两者可靠性不在同一级别——非 OpenClaw 路径的评审仍可能被执行记忆污染。

### 闭环验证模型选择与怀疑论提示

> [软约束·全平台] prompt 级建议——Agent 可能不执行

**规则 A — 模型分离**（解决「视角污染」）：

```
闭环验证模型选择（优先级递减）：
1. 最优：与执行阶段不同的模型（执行 DeepSeek → 验证 GLM/Claude）
2. 可接受：同模型但不同 session（OpenClaw session.spawn 工程隔离）
3. 最低底线：同模型同 session + 重新 Read task/logs（非 OpenClaw 当前路径）
⛔ 禁止：同模型 + 凭执行记忆评审（零隔离）
```
> [软约束·全平台] prompt 级建议——Agent 可能不执行

**规则 B — 怀疑论提示**（解决「指令污染」）：

闭环验证的指令必须独立编写，不能复用执行阶段的指令。评判器提示应：
- 明确告知「这是别人执行的产出，你的任务是找问题」
- 要求逐项对照验收标准检查，不笼统问「有没有问题」
- 包含「列出至少 2 个潜在风险」类强制批判性要求

> 模型分离解决「视角污染」，怀疑论提示解决「指令污染」——即使换了模型，如果指令和执行时一样，评判器仍会顺着执行逻辑走。两条规则互补。
>
> 规则 A 分级递降（有条件走最优）。规则 B 零成本，所有平台都必须做。

### 输入与流程

输入（区分评审依据 vs 历史参考）：
- `task/logs/`：本次执行痕迹 → **评审主依据**，评审者只认这个
- `scoring/_index.md` / `orchestrator/`：历史沉淀的决策数据 → **仅作参考**，供评分时对齐历史基线，不计入本次评审

> 边界：scoring/orchestrator 是跨任务的历史数据，不含本次执行痕迹，可以读；本次执行痕迹只看 task/logs，不从对话历史里找证据。
> ⚠️ **scoring 追加式写入**：评分写入 `scoring/_index.md` 时**追加新条目**（带日期+任务名的小节），**不覆盖**历史——历史评分是对齐基线和 A/B 对比的依据。格式：`## YYYY-MM-DD · 任务简述` 下接九维表。

→ **5 项结构化 checklist**（复盘时必须逐项打勾，每项附带证据，不依赖直觉）：

| # | 检查项 | 判定标准 | 证据要求 |
|:--:|------|------|------|
| ① | **代码通过** | 测试/lint exit 0 / 构建成功 | 贴 exit code 或最后一行输出 |
| ② | **测试覆盖** | 新增/修改代码有对应测试 | 列出测试文件路径 |
| ③ | **文档更新** | 如有新增/变更接口，文档已同步 | 列出更新的文档文件 |
| ④ | **副作用清理** | 临时文件/测试数据已删除 | `ls` 确认临时目录为空 |
| ⑤ | **规则遵守** | 铁律#1-#10 逐条自检无违规 | 列出有违规风险的铁律编号及自检结论 |

> ⛔ checklist 不是「是/否」勾选——每项必须附带证据。没有证据的「是」= 没有检查。

→ **Diagnosing Box 四维度排查**（失败复盘时使用，按优先级依次排查，不凭直觉猜原因）：

| 维度 | 排查问题 | 排查方法 |
|------|------|------|
| **输入** | prompt 是否模糊？数据是否缺失？上下文是否被截断？ | 回溯原始用户消息，检查 `{SOFAGENT_DATA}/task/logs/` 中的输入记录 |
| **环境** | 权限不足？依赖版本不匹配？磁盘空间不足？ | 检查 `command -v`、文件权限 `ls -la`、`df -h` |
| **工具** | 工具不可用？输出异常？返回值不符合预期？ | 单独运行工具的最小示例，对比预期和实际输出 |
| **模型** | 模型能力不足？幻觉？推理链断裂？ | 用更简单的同类任务测试，对比不同模型的表现 |

→ **反思**（对照 3 问写 think.md，≤200 字）：
  ① 做对了什么？（≥1 条，带外部证据标记）
  ② 做错了什么？（≥1 条，带失败模式标签）
  ③ 下次改什么？（≥1 条，格式参考：做什么 + 怎么做）

→ **外部信号校验**（反思写入前执行）：
  检查本次任务是否有客观证据支持复盘判断：
  - 测试通过（exit code 0）/ lint 无报错（exit code 0）/ API 返回 2xx → 标记 [已验证]
  - 用户明确确认「完成了」「没问题」→ 标记 [用户确认]
  - 无上述证据，纯基于文件推断 → 标记 [LLM自评]

  ⛔ 外部证据判定只接受二值结果——lint 的 exit code、API 的 HTTP status、测试的 pass/fail。

  写入 think.md 时，标记位跟在反思摘要后：`[LLM自评:7.5]` 或 `[已验证]` 或 `[用户确认]`。
  下次加载时，[LLM自评] 条目权重 ×0.3，[已验证] 和 [用户确认] 权重 ×1.0。

  > ⚠️ 以下反思均为 LLM 自评，可能存在确认偏误。反思区的结论应视为「有待验证的假设」，而非「已被证实的事实」。

→ **记忆写入规则**（反思写入 think.md 前执行）：

  写入前 Agent 必须自问——「如果是上周的我，看到这条能避免错误吗？」答不出 → 不写。

  只在满足以下条件之一时写入 think.md 反思区：
  - 同一错误 session 内重复 ≥2 次
  - 导致可验证破坏性后果（exit code ≠0 / 文件误删 / API 4xx）
  - 用户明确说「记住这个」

  不满足以上条件 → 不写。**宁可少记，不可记错。**

→ **记忆遗忘规则**：
  - 30 天前的反思条目 → 降权 ×0.3（从原始权重再打折）
  - 60 天前的反思条目 → 移到 `{SOFAGENT_DATA}/think.archive.md`，不再进入加载链
  - 每 7 天或 think.md 超 5K token 时触发压缩——见 `{OPENCLAW_SCRIPTS}/compress-memory.sh`

> **人类抽样审计**：每完成 10 次闭环反思（task/logs 记录数为 10 的倍数时），
> 在本次反思中附加 `[待人类审计]` 标记，并在对话中提醒用户：
> 「你已经跑了 10 次任务，建议看一眼最近的反思有没有偏。跑偏了告诉我，我会记住。」

→ **评分**：九维加权平均（编排/匹配/经济/流畅/完整/复用/合规/Loop/判断力）→ 前5次只记录不判断

  > **冷启动基准线报告（前 5 次任务）**：task_count ≤ 5 时，closure 模式额外输出以下报告。不需要精确计算数值——关键是告诉 Agent「前 5 次任务应该输出这个报告，这是后续所有评分的参照系」。
  >
  > ```
  > ## 冷启动基准线报告（前 5 次任务）
  > 
  > | 维度 | 初始值 | 解读 |
  > |------|:------:|------|
  > | 任务完成率 | X.X | 子任务完成数 / 子任务总数 |
  > | 编排准确性 | X.X | ao compose / 默认编排拆解是否合理 |
  > | Skill 匹配度 | X.X | 分配的角色/技能是否合适 |
  > | 模型经济性 | X.X | Flash/Pro 分配是否合理 |
  > | 执行流畅度 | X.X | 有无卡顿/重试/超时/回滚 |
  > | 结果完整性 | X.X | 用户要的东西做全了没有 |
  > | 复用潜力 | X.X | 同类任务模板通用性 |
  > | 流程合规 | X.X | 是否跳步/绕过了必须的检查点 |
  > | Loop 有效性 | X.X | 检查点是否真正起到了作用 |
  > | 判断力 | X.X | 该停就停（弃权率） |
  > 
  > > 这是你的 Agent 在九维评分上的初始基线。之后的每次复盘会更新这些值——这个基线是参照系。
  > ```

  **第九维 · 判断力（弃权率）**：
  - 拒绝 1 次高风险任务（输出 `[准入检查: REJECT]`）→ +0.5 分
  - 不该做但做了（跳过准入检查直接执行高风险任务）→ -1.0 分
  - 权重初始 0.1（样本不足），随数据积累调高
  - ⛔ 判断力和生产力（前八维）分开计分——不放在同一个总分里。「你很能跑」和「你很会判断什么不该跑」是两件事
  - LLM 自评判断力维度时，降权 ×0.3（同其他 LLM 自评条目）

→ **A/B**：同类≥3次对比 → 沉淀最优拆法

→ **汇报**：口头总结完成了什么、改了什么、下一步

> 💡 运行 `bash sofagent/scripts/verify-evidence.sh` 检查本轮评分是否有客观证据支撑。
