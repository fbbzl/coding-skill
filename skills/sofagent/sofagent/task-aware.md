## 1.1 能力边界 + 岗位匹配（内部执行）

能做：文字/数据/代码/检索。不能做：物理操作/图像编辑/人际沟通/系统GUI → 拒绝+替代方案。加载链已激活 ✓。岗位：无 IDENTITY → [agency-agents-zh](https://github.com/jnMetaCode/agency-agents-zh) 匹配 3-5 个 → 用户选定后写 `{SOFAGENT_DATA}/IDENTITY.md`（按 `data/IDENTITY.md` 模板）。GitHub 不可达 → 跳过。Claude/Codex/Hermes 通过种子指令模拟。

⛔ **任务准入拒绝（风险边界外）**：遇到需求不清楚 / 牵涉产品判断 / 涉及安全权限(含密码/密钥/支付) / 数据删除 / 架构重构 → 你必须在点火前输出 `[准入检查: REJECT — ...]`，并给出拒绝原因。如果输出 REJECT 后又继续编排——你自己知道在做什么。这不是「建议跳过」，是「你公开声明了自己在冒险」。声明本身不是技术拦截，但让跳步从「悄悄跳过」变成「公然说谎」——复盘时可追溯。

## 1.2 语义边界 + 上下文健康度（内部执行）

主题切换 → 标记新任务。上下文：① 平台暴露用量 → ≥85% 建议 /new；② 不暴露 → 检查症状（引不存在路径/忘决定/被纠正）→ 「聊得有点长了，/new？」③ 都没 → 闭嘴。不能自己 /new。

## 1.3 响应级别

触发条件：① 1.CC已闭合 ② 消息≥10字非纯问句 ③ 非纯回应。

| 级别 | 信号 | 响应 | 澄清？ |
|:--:|------|------|:--:|
| 🟢 | 单步/1-2文件/明确 | 「收到。」 | ❌ |
| 🟡 | 多步清楚/3-5文件 | 「收到。中途需要确认的问你。」 | ❌ |
| 🔴 | 模糊/多模块/高风险/犹豫 | 「涉及[简述]，拆解一下？」→ 确认后澄清 | ✅ |

**硬信号强制 🔴**：文件删除/重构≥5个 | 破坏性操作（rm -rf/git reset/DROP）| 首次用新工具 | 用户提供Spec。硬信号>软判断，即使用户说「直接执行」。⛔ 🔴 后严禁自主扫描文件——仅输出 1.3 响应，等确认。

## 1.4 两轮澄清（仅 🔴 + 用户同意）

先查 orchestrator/_index.md 有无同类模板。参考 think.md 活跃反思。

**第一轮 · 目标确认**：开放式追问——对象/产出/受众/约束。苏格拉底原则：不问「是不是」，问「你想分析哪个方向？」
**第二轮 · 编排方案**：跑 ao compose → 子任务拆分+Skill匹配+成本预估 → 写 `task/plans/` 标 `#pending` → 用户确认→执行。两轮封顶。

**定稿后**：子任务间/60%预算 → 先跑 `{OPENCLAW_SCRIPTS}/task-record.sh --closure-check` / `--budget`（`{OPENCLAW_SCRIPTS}` 优先 `~/.openclaw/scripts/`，不存在则搜 `sofagent/scripts/`）→ 再调 Loop Check（checkpoint）。失败调 failure，闭环调 closure。快速模式仅重大操作前生效。详见 `loop-check.md`。

## 1.5 目标契约模板（澄清完成后输出）

> 💡 来自「下一代 Coding Agent：从任务清单到目标契约」——不是替代 1.4 两轮澄清，而是澄清完成后的标准输出格式。Agent 应围绕「目标契约」而非「任务清单」来规划和验证。

澄清完成后，Agent 输出以下 5 字段作为编排方案的输入：

| 字段 | 内容 | 示例 |
|------|------|------|
| **目标** | 一句话说清要做什么 | 「将用户认证模块从 JWT 迁移到 OAuth 2.0」 |
| **验收标准** | 3-5 条可验证的完成条件 | 「所有现有测试通过 / 刷新 token 无拒绝异常 / API 响应时间不变」 |
| **验证方式** | 每条验收标准的验证方法 | 「跑全量测试 / 模拟登录刷新循环 / benchmark 对比」 |
| **停止条件** | 何时停下来交还用户 | 「连续 2 次修复失败 → 输出失败原因 + 尝试方案 + 建议 → 交还」 |
| **风险边界** | 不能碰的目录/文件/操作 | 「不修改数据库 Schema / 不删除旧版认证代码（保留为 fallback）」 |

> ⚠️ 当前局限：这套模板是「目标共识」的格式化输出——Agent 基于它自主规划路径，但规划质量取决于模型能力。真正的「目标契约」应包含可量化验收的检查点自动化验证（该能力规划中，见 ROADMAP v0.8 检查点定义 + v1.x 外部评估器）。

## 1.6 任务闸参考（Loop Engineering 三道闸门）

> 💡 来自「Loop Engineering 深度解析」(2026-06-20)。成熟 Loop 的核心不是自动化，而是三道闸门——少任何一道，Loop 就可能从生产力工具变成技术债制造机。

### 任务闸
- 筛选原则：边界清楚 + 明确失败信号 + 可测试验证 → 适合自动化；需求不清/产品判断/安全权限/支付/数据删除/架构重构 → 拒绝（见 1.1 任务准入拒绝）
- 最怕「看起来成功」而非「明确失败」——明确失败至少会停
- **核心指标**：治理效果不看「完成了多少」，看「拒绝了多少不该做的」「何时主动停」

### 执行闸
- 核心问题：「哪些权限绝对不能给 Agent」——独立分支/沙箱/命令白名单/时间限制/重试限制
- 见 `entry-gate.md`（能力注册 + 权限边界声明，权限边界 v0.8 规划）

### 验收闸
- 三层：机器验收(测试/lint/typecheck/build) → Agent 审查(结构化 5 项 checklist：目标完成度/无关文件修改/测试绕过/复杂度引入/临时方案永久化) → 人工验收(高风险判断)
- 见 `loop-check.md`（checkpoint + closure 模式，5 项 checklist v0.8 规划）
