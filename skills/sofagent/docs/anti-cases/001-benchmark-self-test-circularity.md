# 反案例：benchmark 自测循环论证

| 字段 | 内容 |
|------|------|
| 日期 | 2026-06-20 |
| 平台 | WorkBuddy |
| 任务 | 首次 benchmark 实测（任务 #3 / #4 / #10） |
| 期望 | benchmark 能客观对比「带 sofagent」和「不带 sofagent」的 Agent 行为差异 |
| 实际 | 「带 sofagent」侧的 Agent 就是被测试的 Agent 自己——同一会话、同一加载链。Agent 声称自己遵守了约束，但无法证明这份声称是约束生效的结果还是模型本身的判断。同时，「不带 sofagent」侧无法实测（同会话无法卸载），只能填「无法自测」 |
| 为什么没拦住 | **benchmark 设计本身没有解决「谁来测」的问题。** 当前设计假设一个中立的第三方 Agent 执行测试，但实际上测试 Agent 也在 sofagent 加载链内。测试结果如「我触发了底线 #2」、「我遵守了铁律 #5」——这些声明来自被测对象自身，而非独立观察者。用一个比喻：让一个戴着眼罩的人自己报告眼罩是否有效 |
| 修了吗 | 未修。这是 v0.72 benchmark 方法论的根本性问题，不是代码 bug，需要在 benchmark 协议层面解决 |

## 详细分析

### 循环结构

```
┌─────────────────────────────────────┐
│ Agent（已加载 sofagent 三层加载链）  │
│                                     │
│  ┌─────┐    执行测试     ┌────────┐ │
│  │约束层│ ◄────────────► │测试任务 │ │
│  └─────┘                 └────────┘ │
│     ↓                               │
│  "我遵守了约束" ←── 自己报告自己    │
└─────────────────────────────────────┘
```

### 为什么这很重要

1. **对比侧缺失**：benchmark 的「带 vs 不带」是核心设计卖点，但「不带」侧在当前测试模式下无法满足。填「无法自测」是诚实的，但也意味着对比失效
2. **自述不可靠**：think.md 已经记录过「Agent 自查不可靠——自证 = 自欺的入口」。benchmark 测试恰好落入了同一个陷阱
3. **约束层测试无法区分信号来源**：Agent 拒绝 `rm -rf /tmp`——是因为底线 #2 生效，还是因为模型本身的安全训练也会拒绝？这在单会话内无法区分

### 可能的解决方向

| 方案 | 描述 | 成本 |
|------|------|------|
| A. 独立测试会话 | 用不同的 Agent 会话跑「带 vs 不带」，测试者只记录外部可观察行为（工具调用序列、回复内容） | 需要两次会话，手动协调 |
| B. 自动化脚本断言 | benchmark.sh 不依赖 Agent 自述，而是检查工具调用日志（是否实际调用了 rm / ffmpeg / 无追问直接操作） | 需要 Agent 日志输出标准化 |
| C. 多模型交叉验证 | 同一 prompt 发给不同模型跑，观察约束是否一致生效 | 人力成本高 |
| D. 接受自测局限性 | 承认 WorkBuddy 单会话自测仅能验证「约束是否被声明」，无法验证「约束是否改变行为」。对比测试留到 OpenClaw hook 环境 | 零成本，但 benchmark 价值打折扣 |

### 对 benchmark 设计的建议

- 「不带 sofagent」侧应在 benchmark 文件中预填为「需独立测试会话」而非「无法自测」，引导测试者另开会话
- 「约束违规次数」指标对单会话自测而言不可靠——应改为「约束声明次数」（Agent 声称引用约束的次数），并明确标注这是 Agent 自述指标，需交叉验证
- 约束层 PASS/FAIL 应基于**可客观观察的行为**（是否实际执行了危险操作），而非 Agent 自己的声明
