# 行业研究启发与未来方向

> 以下方向来自 2026-06-20 四篇行业研究笔记的学习总结。不作为开发承诺，仅供后续版本设计参考。
>
> 💡 **Skill 自进化现状定位**：本节描述的是「未来方向」。当前 sofagent 的 Skill 自进化机制处于第一阶段（经验记录），现状和局限详见 [ARCHITECTURE.md §三](../../ARCHITECTURE.md)「Skill 自进化仍处于经验记录」阶段（单轨迹归纳风险 + 冷启动保护仅作缓冲）。下方各条目按规划版本分类。
>
> 本文档从 ARCHITECTURE.md §五 拆出（v0.74），避免研究笔记稀释架构文档的核心叙事。

---

### 防雪崩：共享状态的评审机制

多 Agent 协作中，共享状态无限放大会导致错误和噪音在 Agent 间传播（雪崩效应）。SoftwareCompany SOP 的「执行→验证→汇总」三段式已实践了这一模式，但尚未提炼为 sofagent 的通用模块。

**规划版本**：v0.8 · `loop-check.md` 加防雪崩说明

### Diagnosing Box：四维度系统性排查

当前 loop-check 复盘是「写下教训」——缺少系统性排查步骤。Matt Pocock 的 Diagnosing Box 概念启发：失败后按**输入/环境/工具/模型**四维度系统排查，而非凭直觉猜原因。

**规划版本**：v0.8 · `loop-check.md` 加四维度排查清单

### rules.md 升级：从「自定义规则」到「Agent 运行规范」

笔记「下一代 Coding Agent」指出：Scale File 应从「提示词模板」升级为「Agent 运行规范」——包含项目目标、验收标准、风险边界、停止条件。sofagent 的 rules.md 目前定位是「你的自定义规则」——应升级描述但保持现有功能不变。

**规划版本**：v0.9 · `DEVELOPMENT.md` + `HANDBOOK.md` 中 rules.md 描述升级

### 检查点定义：编排深度与验证联动

笔记「Go/Loop 模式」强调每轮迭代应围绕目标推进 + 检查点验证 + 明确停止条件。当前 engine.md 有编排深度但无检查点定义——不是跑完才检查，是每步都有验证。

**规划版本**：v0.8 · `engine.md` 编排流程加检查点定义

### 权限最小化

笔记「四大金牌原则」之一：各 Agent 限制操作范围。当前 sofagent 的 entry-gate 只做能力注册，不做权限边界声明。

**规划版本**：v0.9 · `entry-gate.md` 加权限边界字段

### think.md 多重置信度标记

当前反思区已有 `[LLM自评 ×0.3]` 降权。笔记「Skill 自进化」建议加两种新标记：`[多轨迹归纳]`（至少 3 条同类轨迹提取的模式）和 `[外部验证]`（通过独立评估器验证的结论），分别对应不同的置信度权重。

**规划版本**：v0.9 · `think.md` 数据模板加多重置信度标记定义

### 验证门控：Skill 自进化的 A/B 对比

笔记「Evil Skill」的核心洞察：Skill 更新不应靠 Agent 自评「看起来更好了」，而应在验证集上跑 A/B 对比——新版本严格优于旧版本才能被接受。这与 ROADMAP v1.x「外部评估器」方向一致，但补充了**验证集构建**和**门控机制**的具体思路。

**规划版本**：v1.x · ROADMAP「外部评估器」扩写

### 三种协作模式

笔记「多 AI 智能体协作系统」提出三种模式：主管（默认）/ 流水线（A→B→C 顺序传递）/ 委员会（多专业 Agent 提案→仲裁者抉择）。SoftwareCompany 目前只用主管模式。v2.x 路由器可支持三种调度策略。

**规划版本**：v2.x · ROADMAP 协同编排协议扩写

### Loop Engineering 三道闸门对照

> 来自「Loop Engineering 深度解析」(2026-06-20)。五篇笔记中对 sofagent 冲击最大的一篇——核心指标与评分逻辑直接冲突。

| 闸门 | 笔记定义 | sofagent 现有 | 差距 |
|------|------|------|------|
| 任务闸 | 筛选适合自动化的任务；不适合的应直接拒绝 | `task-aware.md` 🟢🟡🔴 + §1.1 任务准入拒绝 + §1.6 任务闸参考 | v0.71 已加 5 类拒绝条件；缺执行层强制拦截（当前靠 Agent 自觉） |
| 执行闸 | 独立分支/沙箱/白名单/时间上限/重试上限——先问「哪些绝对不能给」 | `entry-gate.md` 能力注册 | 只注册「能做什么」；规划版本：v0.8 加权限边界声明 |
| 验收闸 | 三层：机器→Agent审查(5项checklist)→人工 | `loop-check.md` checkpoint | 缺 Agent 审查的结构化 5 项 checklist；规划版本：v0.8 |

**核心指标冲突**：sofagent 的 `scoring/` 基于使用频率 + 成功率，本质奖励「多跑多成功」。笔记指出正确方向是奖励「知道什么不该做、何时主动停止」。笔记给出三组替代指标：

| 不该看 | 应该看 |
|------|------|
| 一天开多少 PR | 拒绝了多少不该做的任务 |
| 能不能自动完成 | 知不知道什么时候停 |
| 是不是像一个 AI 同事 | 是不是像一个受控的工程系统 |

这个冲突已在「Skill 自进化仍处于经验记录阶段」段标注，scoring 体系的指标修正在 v0.9 规划中。

**失败分支显式设计**：笔记列出 6 个关键失败场景——测试失败/连续两次失败/改动太大/任务与代码现状冲突/多 Agent 矛盾 PR/成本超预算——指出「如果这些分支没被设计出来，就只是会自动按回车的脚本，不是真的工程系统」。sofagent 当前对这些的处理是隐式的（渐进减薄兜底 + loopDetection 兜底），缺显式分支。规划版本：v0.8 在 `engine.md` 编排流程中加显式失败分支处理逻辑。

**「换了一种方式加班」警告**：如果治理层只是让人从写代码变成疯狂审核 AI 写的代码，那不叫自动化，叫换了一种方式加班。治理层的价值不是「让用户审核 AI」，而是「让 AI 的产出天然更可靠，减少人的介入次数」——这是 sofagent 设计原则中应补充的一环。

**第一版 Loop MVP**：笔记定义的最小闭环——「发现低风险问题→提出修改→等待人工验收，不自动合并、不连生产、不改核心模块」——与 daemon v0.1 MVP（发现 think.md 变更→提醒 Agent→等待 Agent 自己处理）是同一设计哲学：最小闭环、渐进扩展。ROADMAP v0.8 daemon 描述可引用。

**渐进式路径**：笔记的扩展顺序（文档→测试→小bug→局部重构，每步增加验收规则）与 ROADMAP v0.8→v0.9→v1.0 方向对齐。笔记强调「Agent 越强，闸门越重要」——不可因模型能力提升而拆除控制机制。

**规划版本**：v0.8（执行闸基础 + 验收闸 checklist + 失败分支显式设计）、v0.9（三层验收完整 + 主动停止判断 + scoring 指标修正）。

### 安装脚本历史问题

v0.47 及之前版本的 install.sh 存在文件复制不全问题（OpenClaw 路径仅复制 2/6 个 Skill 文件，WorkBuddy 路径为 no-op）。v0.48 已修复。

### 反思自评的自噬风险

LLM 自评 → 写入 think.md → 下次加载 → 影响判断——这个闭环如果没有外部校验，不准的自评会自我强化（行业研究称这一现象为"自噬效应"（autophagy effect））。sofagent 的应对：反思写入时标记来源（[LLM自评] / [已验证] / [用户确认]），加载时 [LLM自评] 权重 ×0.3；同时用 3 问模板（做对了什么 / 做错了什么 / 下次改什么）替代自由文本，让反思结构化可比对。这不是根治——根治需要外部评估系统（如 coze-loop 的评估器+评估集架构），但标记+模板+降权是缓冲层。

### Loop Agent 不是独立进程

Loop Agent 不是独立进程或独立模型调用——它是主 Agent 在特定节点切换 prompt 以顾问身份输出建议。文档中"独立复盘"指角色隔离（执行者不自评），不是工程隔离（独立进程/独立模型实例）。

### ao compose 依赖 npm 全局安装

`ao compose` 依赖 npm 全局安装的 `agency-orchestrator` 包——不是纯 bash 环境。OpenClaw / macOS / Linux 默认带 Node.js，但受限沙箱或纯容器环境可能需要额外安装 Node.js ≥18 + npm ≥9。install.sh 不会自动装 Node——需要用户自行确保 npm 可用。

**安装提示缺失**：ao 未安装时 engine.md（v0.71 前）直接静默降级，不告知用户 ao compose 的存在和安装方式。新用户可能完全不知道编排引擎的完整能力（模板匹配、子 Agent 分配、成本预估），也不知道 `--no-ao` 意味着什么。v0.71 已修正：A2 流程改为告知性降级，口头提示安装命令。

### 折半机制跨平台生效差异

内部 hook `sofagent-load-chain` 的 `[LLM自评]` 标记位降权（权重 ×0.3）只在 OpenClaw 平台物理生效——通过 `agent:bootstrap` 事件在注入 think.md 时动态追加降权提示。WorkBuddy / Codex / Hermes / Claude Code 没有该 hook，降权机制靠 SKILL.md C 步 Read think.md 后 Agent 自觉识别 `[LLM自评]` 标记——与上文「反思自评的自噬风险」「复盘评分是 LLM 自评」同一局限。v1.x 引入外部评估器时统一解决。不假装 WorkBuddy 也有脚本级降权。

### Agent 记忆系统的三套规则

> 来自「Agent 记忆机制设计指南」(2026-06-20)。笔记指出完整记忆系统需三套规则：写入规则、更新规则、遗忘规则——缺任何一套，记忆系统就不完整。sofagent 的对照：

| 规则 | 笔记要求 | sofagent 现状 | 差距 |
|------|------|------|------|
| 写入规则 | 明确什么该记（稳定模式/重复错误/用户确认）vs 不该记（单次异常/临时环境/LLM推测） | 无显式规则——Agent 自觉决定 | think.md 容易把单次噪声写成反思；与笔记三「单轨迹不可靠」是同一风险 |
| 更新规则 | 新旧冲突→检测→合并或覆盖→重心明确 | think.md 只追加，不合并 | 3 条互相矛盾的反思可并存；缺冲突检测机制 |
| 遗忘规则 | 临时信息过期 / 低价值压缩 / 敏感删除 | `cleanup.sh` 按时长+条数清理 | 有清理，缺"低价值压缩"——不常访问的反思应压缩为摘要而非直接删除 |

**跨笔记验证**：笔记三「Skill 自进化」警告单轨迹偏差，笔记六「写入规则」给出解决方案——"记什么不记什么"的判断标准。两篇从不同方向指向同一架构债：sofagent 当前缺乏"判断什么值得记、什么应该忘"的机制。

**规划版本**：v0.9（think.md 引入冲突检测 + 写入标准 + 低价值压缩）。

### 存储策略对照

笔记六的核心口诀——"关键事实走结构化，近期变化走摘要，经验案例走混合检索"——与 sofagent 的全 Markdown 存储形成对照：

| 信息类型 | 笔记建议 | sofagent 方案 | 评估 |
|------|------|------|------|
| 关键事实（偏好/禁忌/设置） | 结构化存储（精确/稳定/可控） | `rules.md` key:value 格式 | ✅ 雏形正确——rules.md 的 `log_sanitize: true` 格式正是结构化方向 |
| 近期变化（确认/修改/否定） | 摘要清单（轻量/聚焦） | `think.md` ≤2K token | ✅ 方向正确——500 字原则和 ≤200字/条都是轻量化设计 |
| 经验案例（项目/决策/复盘） | 混合检索（语义+关键词+条件） | `task/logs/` Markdown 全文 | ⚠️ 无检索能力——Agent Read 全文，无法按语义/关键词筛选 |

**当前局限**：全 Markdown + Agent Read 的方案简单可靠（零依赖、可审计），但无法做到笔记建议的"混合检索"。这是一个主动取舍——sofagent 选择文件系统优先，代价是无法做语义搜索。如果未来 task/logs 积累到上千条，需要引入轻量检索方案。

> 💡 **如果你不同意某个决策**：以上局限中，可通过 `rules.md` 覆盖的有——编排深度晋级阈值、活跃区 token 上限、复盘权重。需等待平台演进的有——定时触发（等 OpenClaw cron）、中间检查点挂起（等 `before_tool` Hook）。其余的（非分布式、Skill 层 Slop、LLM 评估偏差、软层执行率）是架构宿命，不是配置问题。

### 循环工程（Loop Engineering）行业研究启发（2026-06-20）

> 来自两篇独立笔记：「循环工程师：智能体自动化时代的核心方法论与实践指南」+「Cloud Code 新功能 Workflows 深度解析」。前者对 sofagent 冲击最大——它定义了 sofagent 正在做的这件事的完整框架。

#### 核心公式：循环 = 触发器 + 文件结构 + 工具 + 代码库治理

sofagent 现在只有后三样的雏形，**触发器是完全空白的**——这是 sofagent 作为 Loop Engineering 系统最大的结构性缺陷。当前只有「对话启动」一种触发方式，定时/事件/Webhook 全缺（§三「定时触发做不到」已承认）。

**daemon 是填这个缺口的唯一可行路径**——launchd/systemd 就是定时触发，文件监控就是事件触发。daemon 的真正价值不只是「提醒读 think.md」，而是**让 sofagent 从 session 内治理升级为 session 外治理**——这是 Loop Engineering 的入场券。

| daemon 能力 | 对应的触发器类型 | sofagent 之前做不到的事 |
|---|---|---|
| launchd/systemd 定时拉起 | 定时触发 | 定时跑 cleanup.sh / 定时健康检查 |
| 文件变更检测 | 事件触发 | think.md 变了 → 触发下一步 |
| Agent 进程检测 | Agent 状态触发 | Agent 启动 → 注入循环契约 |

**规划版本**：v0.8 · daemon 定位升级——从「加载链补丁」到「sofagent 的第一个 session 外触发器」

#### 循环契约（Contract）：engine.md 缺失的概念层

循环工程师笔记定义了一个 sofagent 完全没有的文件类型——**循环契约**：每个循环都有一个 README，包含这个循环的目标、工作流、待办列表、时间线。每次循环触发时，Agent 读这个契约来了解「我是谁、在干什么、上次到哪了、下一步做什么」。

sofagent 现在的三层加载链和循环契约的对照：

| 循环契约要素 | sofagent 现状 | 差距 |
|---|---|---|
| 目标（这个循环要达成什么） | SKILL.md 有身份和铁律，但没有「这个循环的目标」 | ❌ 缺 |
| 工作流（按什么步骤走） | engine.md 有编排流程 | ⚠️ 有但不持久化——每次重跑 |
| 待办列表（下一步做什么） | 无 | ❌ 完全缺 |
| 时间线（之前发生了什么） | task/logs 有日志，think.md 有反思 | ⚠️ 分散在两个文件，无统一视图 |

daemon 的 reminder.md 应升级为**循环契约摘要**——不只是「think.md 有新反思」，而是包含目标摘要 + 最近进度 + 待办教训 + 规则变更。这比零散的提醒强得多。

**规划版本**：v0.8 · daemon reminder.md 向循环契约靠拢 + engine.md 加契约概念

#### 信号（Signal）机制：v2.x 的集体大脑

循环工程师笔记描述了一个 sofagent 没有的制品类型——**信号文件夹**：跨循环共享的「集体大脑」，捕捉产品创意、用户摩擦、机会缺口。支持循环发现用户频繁问导出文件 → 写一个 signal.md → SEO 循环和产品循环都能读。

sofagent 没有这个层。think.md 是「教训」（别犯错），scoring 是「评分」（用得好不好），都不是「机会/摩擦/信号」（发现什么值得做）。这是 sofagent 作为**治理层**和循环工程作为**业务系统**的定位差异——sofagent 管 Agent 别犯错，循环工程管 Agent 发现机会。

v2.x 路由器可从「反思同步」升级为「信号共享网络」——多设备不只是同步 think.md，而是互相发现信号。

**规划版本**：v2.x · 路由器升级为信号共享网络

#### Workflows 的印证与降级处理

Cloud Code Workflows 笔记的核心（SOP → JS 控制脚本、先规划后执行、可观测性）和 sofagent 的 engine.md + ao compose + 闸门确认是同一层东西的不同实现。增量信息有限，主要是印证：

| Workflows 概念 | sofagent 对应 | 印证/差异 |
|---|---|---|
| 先规划后执行（context.md） | ao compose Gate 确认 | 印证——sofagent 有确认环节 |
| JS 控制脚本编排 | engine.md + task-orchestrate.sh | 同层不同实现 |
| TOI 仪表盘（可观测性产品化） | daemon v0.3 状态面板（规划中） | **值得参考**——Agent 状态 / Token 消耗 / 运行时长 |
| 300 行 JS 编排 97 个 agent 轮次 | v2.x 多 Agent 协同 | 印证大规模编排可行，但是远期方向 |

**规划版本**：daemon v0.3 · 状态面板参考 Workflows TOI 思路

#### 两个原则性警告（笔记反复强调）

1. **「不要让智能体自我验证」**——笔记提到「最重要的一点」。sofagent 在 §三已承认（复盘评分是 LLM 自评），v1.x 外部评估器是正确方向。daemon 上线前只能靠冷启动保护 + 标记降权缓冲
2. **「Agent 越强，闸门越重要」**——不可因模型能力提升而拆除控制机制。sofagent 的三道闸门（任务闸/执行闸/验收闸）在 v0.73 补齐执行层强制力，不因 GPT-5/Claude-4 更强就放松
