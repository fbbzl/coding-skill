## P0 — 安装/部署断裂（4 条）

### 要解决什么

v0.82 五平台实测发现：**三个平台装完只有宪法没有脚本**。OpenClaw 全新环境 Hook 注册会生成空 .tmp 覆盖配置。一个 set -u 未定义变量炸弹藏在 REMOTE_MODE 分支。

### 修复结果

| # | 问题 | 根因 | 修复方案 | 验证 |
|:-:|------|------|---------|:----:|
| 1 | OpenClaw Hook 注册生成空 .tmp | jq 读空文件输出空 .tmp，mv 覆盖导致配置变空 | jq 处理前确保文件存在且非空；.tmp 输出后检查非空再 mv | ✅ |
| 2 | WorkBuddy scripts/ 未部署 | 脚本部署逻辑被包裹在 OpenClaw-only 的 Step 6 块中 | 提取为 Step 5b，对所有平台执行 | ✅ |
| 3 | Claude Code scripts/ 未部署 | 同上 | 同上 | ✅ |
| 4 | install.sh TARGET_DIR 未定义 | REMOTE_MODE 分支引用从未定义的 `$TARGET_DIR`，set -u 下爆炸 | 改为 `$REMOTE_TMP`（第 59 行已定义） | ✅ |

**P0-1 技术细节**：

修复前的代码路径（install.sh 原 Step 6 Hook 注册块）：
```
openclaw.json 不存在
→ jq '...' "$HOOK_CONFIG" > "${HOOK_CONFIG}.tmp"  # 读空文件，输出空
→ mv "${HOOK_CONFIG}.tmp" "$HOOK_CONFIG"            # 空文件覆盖
→ 配置变空，Hook 注册失败
```

修复后双重防线：
```bash
# 防线 1：确保文件存在且有有效 JSON
[ -f "$HOOK_CONFIG" ] || echo '{}' > "$HOOK_CONFIG"
[ -s "$HOOK_CONFIG" ] || echo '{}' > "$HOOK_CONFIG"

# 防线 2：jq 输出后检查非空再 mv
jq '...' "$HOOK_CONFIG" > "${HOOK_CONFIG}.tmp" 2>/dev/null && \
  [ -s "${HOOK_CONFIG}.tmp" ] && mv "${HOOK_CONFIG}.tmp" "$HOOK_CONFIG" && REGISTER_OK=1
```

**P0-2/3 架构调整**：

```
修复前（Step 6 被包裹在 OpenClaw-only 块中）：
  Step 5: 部署 Skill 文件（所有平台）
  Step 6: Hook 部署 + 脚本部署 + 数据目录（仅 OpenClaw）  ← 脚本也被锁在里面
  Step 7: 断路器配置（仅 OpenClaw）

修复后：
  Step 5:  部署 Skill 文件（所有平台）
  Step 5b: 部署配套脚本 + 数据目录（所有平台）  ← 新提取的公共步骤
  Step 6:  Hook 部署（仅 OpenClaw）
  Step 7:  断路器配置（仅 OpenClaw）
```

## P1 — 体验断裂 + 代码加固（3 条）

### 要解决什么

三个"不影响安装但影响体验"的问题：手动平台种子指令要用户自己粘贴、daemon-status 在 `--detect` 时报 `local` 错误、daemon-lib 写 JSON 后不校验是否损坏。

### 修复结果

| # | 问题 | 根因 | 修复方案 | 验证 |
|:-:|------|------|---------|:----:|
| 5 | CLAUDE.md 种子指令不自动写入 | claude/codex/hermes 只 echo 打印，不写文件 | 按 platform 确定目标文件，查重后自动追加写入 | ✅ |
| 6 | daemon-status.sh `local` 函数外使用 | `--detect` 的 else 分支在 case 块（脚本顶层）用 `local`，set -e 下报错退出 | 删除 `local` 关键字（顶层变量本就是全局） | ✅ |
| 7 | daemon-lib.sh JSON 写入无校验 | `set_json_field()` 用 sed -i 修改后直接 return 0，不校验 | 末尾增加大括号配对校验（纯 bash 零依赖方案） | ✅ |

**P1-5 附带修复**：

种子指令写入逻辑中发现 `ok "种子指令已存在于 $SEED_FILE，跳过写入"` 里 `$SEED_FILE` 后跟全角逗号 `，`，bash 将 `SEED_FILE，` 解析为变量名导致 unbound variable。改为 `${SEED_FILE}` 大括号定界解决。

> 这是项目 BSD/macOS 兼容教训库中「全角括号紧邻变量是炸弹」的又一个实例。

**P1-7 设计决策**：

daemon-lib.sh 的设计原则是**零外部依赖**（纯 bash，不依赖 jq）。因此 JSON 校验不用 `jq -e` 而是用逐字符大括号计数：

```bash
local brace_open=0 brace_close=0
while IFS= read -r -n1 ch; do
  [ "$ch" = "{" ] && brace_open=$((brace_open + 1))
  [ "$ch" = "}" ] && brace_close=$((brace_close + 1))
done < "$DAEMON_JSON"
if [ "$brace_open" -ne "$brace_close" ]; then
  return 1
fi
```

> 不解析完整 JSON 语法（那需要 jq），只查大括号配对。对 daemon.json 的扁平结构足够。迁移到 jq 的触发条件已在文件顶部注释（字段数 >10 或出现嵌套对象）。

### 要解决什么

engine.md 的 ao 能力探测用文字描述规则，不如决策表直观。ARCHITECTURE.md 致谢表的 arXiv 链接有实质错误。CI 缺 shellcheck lint。

### 修复结果

| # | 问题 | 修复方案 | 验证 |
|:-:|------|---------|:----:|
| 8 | engine.md A2 节伪代码描述 | 新增 3 行快速决策表（保留原有详细话术） | ✅ |

| 10 | 缺 shellcheck CI | 新建 `.github/workflows/shellcheck.yml`，advisory 模式 | ✅ |

**P2-9 技术细节**：

致谢表两条相邻引用，arXiv 号张冠李戴：
- SkillOS（Google Cloud AI Research）→ arXiv 2605.06614 ✅（原本就正确）
- SkillOpt（Microsoft Research）→ 原链接 2605.06614 ❌（这是 SkillOS 的号）

通过 arXiv 官网核实，SkillOpt 的正确号为 2605.23904。修正后两条不再冲突。

**P2-10 CI 设计**：

```yaml
# advisory 模式——先跑起来产出报告，暂不阻塞合并
severity: warning  # 只报 error + warning，不报 style/info 噪音

# 屏蔽 5 类规则（项目当前代码风格，全改成本高）：
SHELLCHECK_OPTS: -e SC2086 -e SC2155 -e SC2034 -e SC1090 -e SC1091
```

| 规则 | 含义 | 屏蔽理由 |
|------|------|---------|
| SC2086 | 双引号缺失 | sofagent 路径不含空格，故意的 |
| SC2155 | declare+assign 遮蔽返回码 | 大量 `local x=$(cmd)` 模式，全改成本高 |
| SC2034 | 未使用变量 | 误报多 |
| SC1090/SC1091 | source 文件无法静态分析 | 动态 source 路径无法静态分析 |

> 等后续版本逐步清理 warning 后再收紧。path trigger 只在 `sofagent/scripts/**/*.sh` 变更时触发，避免无关 PR 跑 CI。
