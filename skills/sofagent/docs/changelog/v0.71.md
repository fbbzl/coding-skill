## 四、ROADMAP 策略重构

> 由开发者与技术 VP 角色三轮深度讨论驱动：第一思（daemon 时机错误）→ 第二思（企业级太薄 + "适配"是吹牛）→ 第三思（诚实调性 + 谦虚内敛 + 幽默）。

### 改了什么

| 维度 | 改前 | 改后 |
|------|------|------|
| **v1.0 条件** | "v0.6x + v0.7x 全部完成" | 条件清单：daemon ≥30天 / ≥3外部用户 / 安装率≥90% / 5平台全覆盖 |
| **daemon 位置** | v1.x（v1.0 发布后） | v0.8（v1.0 之前——没有外部强制力不配叫 v1.0） |
| **版本路径** | v0.7x → v1.x → v2.x | v0.8（daemon+兼容性）→ v0.9（企业级+Beta）→ v1.0 |
| **企业级** | 零散提了几句加密/Docker | 独立维度：age加密 / 脱敏增强 / 审计报告 / 多用户 / Docker / 团队部署 |
| **平台策略** | "支持五大平台" / "适配" | "提升兼容性" / 产能力矩阵（✅⚠️❌❓）——诚实标注实测/未测 |
| **Beta 阶段** | 无 | v0.9 Beta：20人·30天公测 + Bug-only迭代 |
| **调性** | 像"AI写的乐观计划书" | 诚实说"不知道"、daemon"只是个爱唠叨的后台进程"、不画饼 |

### 设计决策

1. **daemon 不在 v1.0 之后**：加载链脆弱性是最大架构债——没有 daemon 的 v1.0 = OpenClaw 专属，不配叫正式版
2. **v0.9 做 Beta**：开源项目直接 v1.0 大概率翻车。Beta 30 天 + Bug-only 是开源社区的成熟模式
3. **诚实 > 乐观**：GitHub 高手两秒就能闻到画饼——与其被嘲笑，不如主动说"我们不知道"
4. **企业级不只是加密**：数据分类/审计报告/多用户隔离/批量部署 比 加密更优先——因为企业先要知道"有哪些敏感数据"
5. **不说"适配"，说"提升兼容性"**：Claude Code/Codex/Hermes 架构与 OpenClaw 根本不同，daemon 只解决"提醒"问题，编排引擎天然不可用——不吹牛

---

## 设计决策汇总

1. **版本号 0.71**：改动量超 patch 级，semver 合理
2. **P0 全修，P1 全修，P2 全修**：共 24 项 + 第三方审查 16 项
3. **AO API Key 不设自动优先级**：用户自行填写环境变量，不做智能选择
4. **目标契约模板放 task-aware.md**：作为两轮澄清后的输出格式，不替代现有流程
5. **Skill 自进化阶段诚实标注**：明确当前处于「经验记录」阶段——TRACE2SKILL/Evil Skill/Skill Opt 均为未来方向
6. **行业研究方向写进 ARCHITECTURE 而非 ROADMAP**：ROADMAP 是给路人看的路线图，ARCHITECTURE 是给深入阅读者看的设计参考

## 已知问题

- 加载链步进脆弱性是架构层面问题，等 v1.x daemon，v0.71 不碰
- `sanitize()` JWT/AWS 脱敏需评估 regex 复杂度
- `cleanup.sh` 概率触发等到 v1.x 改定时
- `handler.ts` 集成测试需 OpenClaw 环境
- **ao compose 安装提示**：v0.71 前静默降级，新用户不知道完整编排能力存在——已修正为告知性降级（D8）
- **Skill 自进化**：当前处于经验记录阶段，scoring/ 基于单次轨迹修改评分——存在单轨迹偏差风险（冷启动保护仅缓冲，未根本解决）
