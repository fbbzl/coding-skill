# v0.73 开发日志

> **为什么公开这个文件**：v0.72 修了门面（README 的谎话 + EVIDENCE 的空架子），但运行时逻辑里的结构性问题一个没碰——think.md 是噪声场、scoring 奖励跑量不奖励判断、编排引擎没有检查点、记忆规则不存在。v0.73 把这些全修了。
>
> **定位**：v0.72 是「让外面的人不再被误导」，v0.73 是「让里面的人不再被自己骗」。

---

## 做了什么

v0.73 = 运行时逻辑加固版本。v0.72 只碰门面（README / EVIDENCE / benchmark.sh + 两个加固项），运行时逻辑全推到 v0.73。

十件事：

1. **任务闸执行层**：🔴 复杂任务点火前必须显式输出准入检查 PASS/REJECT
2. **执行闸权限边界**：entry-gate 加权限边界声明——不只说能做什么，还要说绝对不能做什么
3. **验收闸 checklist**：loop-check 从自由文本复盘升级为 5 项结构化检查 + 四维度排查 + 防雪崩说明
4. **记忆系统最小闭环**：think.md 终于有了写入/更新/遗忘三套规则
5. **scoring 判断力维度**：不再只奖励跑量，把「知道什么时候该停」算作正向分
6. **编排引擎加固**：每步加验证节点 + 6 个显式失败分支 + 停止条件
7. **rules.md 升级**：从「自定义规则」升级为「Agent 运行规范」
8. **安装门槛降低**：从 4 步到 1 步——打通 ClawHub/SkillHub 一 URL 安装
9. **ROADMAP ASCII → Mermaid**：ASCII 图在移动端塌了，换成 Mermaid
10. **constitution/ 扁平化**：`constitution/rules.md` → `rules.md`——用户改规则不用钻两层目录

## 一、任务闸执行层（P1）

> 评审原话：「task-aware §1.1 有准入检查但没有强制力——Agent 可以跳过准入直接进入编排。」

### 问题

| # | 评审发现 | 严重程度 |
|:--:|------|:--:|
| T1 | **Agent 可跳过 task-aware §1.1 准入检查**：5 类高风险任务（需求不清/产品判断/安全权限/支付/数据删除/架构重构）的拒绝声明只是文字，没有流程强制力 | P1 |
| T2 | **Agent 越强闸门越不重要** — 这是个认知陷阱：强模型不需要治理=不成立了，反而越强的模型越能在「钻空子」上发挥创造性 | P1 |

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| T1-A | **engine.md 点火前加硬检查**：🔴 复杂任务点火前，Agent 必须显式输出一行 `[准入检查: PASS]` 或 `[准入检查: REJECT — {原因}]`。不输出这行 = 不走编排。这不是技术拦截（prompt 层做不到），是**流程钉**——Agent 自己声明通过了检查，复盘时有据可查 | `sofagent/engine.md` | 修改 |
| T1-B | **task-aware.md §1.1 风险边界描述从声明改承诺**：当前写「以下 5 类任务直接拒绝」。改为「以下 5 类任务你必须在点火前输出 `[准入检查: REJECT — ...]`，并给出拒绝原因。如果输出 REJECT 后又继续编排——你自己知道在做什么。」 | `sofagent/task-aware.md` | 修改 |

> 💡 **设计决策**：不试图用 prompt 做技术拦截——因为不可能。改为让 Agent 自己公开声明「我通过了准入检查」。声明本身不是拦截，但让跳步从「悄悄跳过」变成「公然说谎」——复盘时可追溯。

## 二、执行闸权限边界（P1）

> 评审原话：「entry-gate.md 只注册了"能做什么"，没有声明"绝对不能做什么"。不同平台的权限边界不同——OpenClaw 能跑脚本，Web 平台不能跑 rm——但当前注册表没有区分。」

### 问题

| # | 评审发现 | 严重程度 |
|:--:|------|:--:|
| E1 | **entry-gate.md 只有能力注册，没有权限边界**：Agent 不知道自己「绝对不能做什么」，只能靠铁律 #2（不执行危险操作）——一条笼统规则覆盖所有场景 | P1 |
| E2 | **平台能力差异未内化到注册表**：注册表不区分 OpenClaw（有 shell）和 Web 平台（无 shell），Agent 需要在每次任务中自行判断 | P2 |

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| E1 | **entry-gate.md 能力注册表加「权限边界」列**：每个能力加一列声明「绝对不能」做什么。例如：Shell 执行能力 → 「绝对不能 rm -rf / / 删除非项目文件 / 修改系统配置」；文件写入能力 → 「绝对不能覆盖 .git/ / 修改 ~/.ssh/」 | `sofagent/entry-gate.md` | 修改 |
| E2 | **平台差异细分**：注册表加平台维度——能力是否可用按平台分列（OpenClaw ✅ / WorkBuddy ⚠️沙箱 / Web ❌），Agent 不再自行判断 | `sofagent/entry-gate.md` | 修改 |

## 三、验收闸 checklist（P1）

> 评审原话：「loop-check.md 是自由文本复盘——Agent 想写什么写什么。复盘质量完全取决于 Agent 那一刻的自觉性。」

### 问题

| # | 评审发现 | 严重程度 |
|:--:|------|:--:|
| C1 | **复盘是自由文本不是结构化检查**：Agent 可以写「本次任务顺利完成」算作复盘——没有 checklist 约束 | P1 |
| C2 | **缺失系统性排查步骤**：Matt Pocock 的 Diagnosing Box 概念——失败后应按输入/环境/工具/模型四维度系统排查，而非凭直觉猜原因 | P1 |
| C3 | **缺失防雪崩机制**：大模型 Agent 的典型事故模式是「自己修自己改、越修越乱」。当前 loop-check 没有检测和阻止这种模式 | P1 |

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| C1 | **loop-check.md closure 模式加 5 项结构化 checklist**：Agent 复盘时必须逐项打勾——① 代码通过（测试/lint exit 0）② 测试覆盖（新增/修改代码有对应测试）③ 文档更新（如有新增/变更接口）④ 副作用清理（临时文件/测试数据已删除）⑤ 规则遵守（铁律#1-#10 逐条自检无违规）。每项不是「是/否」，必须附带证据 | `sofagent/loop-check.md` | 修改 |
| C2 | **加四维度排查（Diagnosing Box）**：失败复盘时，增加「输入（prompt 模糊？数据缺失？）/ 环境（权限？依赖版本？）/ 工具（工具不可用？输出异常？）/ 模型（模型能力不足？幻觉？）」四维度系统排查，不凭直觉 | `sofagent/loop-check.md` | 修改 |
| C3 | **加防雪崩说明**：共享状态（文件系统、git working tree、环境变量）被多个子任务并发修改的风险——要求子任务间显式标注共享状态变更，违者 checkpoint 强制暂停 | `sofagent/loop-check.md` | 修改 |

## 四、记忆系统最小闭环——三套规则（P1）

> 评审原话：「记忆系统有四层结构但缺失三套核心规则。think.md 的写入没有标准、只追加不合并、永不过期——一条错误记忆的破坏力大于十条正确记忆的防护力。」

### 问题

| # | 评审发现 | 严重程度 |
|:--:|------|:--:|
| M1 | **缺失写入规则**：think.md 没有「什么是值得记录的教训」的判断标准。Agent 把单次异常当成永久规律，噪音污染上下文 | P1 |
| M2 | **缺失更新规则**：只追加不合并。第 3 次任务修正了第 1 次的误判，但两条矛盾条目共存——Agent 不知道该信哪个 | P1 |
| M3 | **缺失遗忘规则**：think.md 永不过期。30 天前的教训已不再适用，仍在占用 token | P1 |

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| M1-A | **新增写入标准**：think.md 反思区只在满足以下条件之一时写入：① 同一错误 session 内重复 ≥2 次 ② 导致可验证破坏性后果（exit code ≠0 / 文件误删 / API 4xx）③ 用户明确说「记住这个」。不满足不写 | `sofagent/loop-check.md` closure 模式 | 修改 |
| M1-B | **新增写前检查**：写入前 Agent 必须自问——「如果是上周的我，看到这条能避免错误吗？」答不出 → 不写 | `sofagent/loop-check.md` | 修改 |
| M2 | **新增合并规则**：每 7 天或 think.md 超 5K token 时触发压缩——同一失败模式标签条目合并为一条，保留最新结论，删除矛盾的旧条目。压缩后备份 `think.YYYY-MM-DD.bak`，**仅保留最近 3 份备份，超出自动删除最旧的** | `sofagent/scripts/compress-memory.sh` | **新建** |
| M3 | **新增遗忘规则**：30 天旧条目降权 ×0.3，60 天旧条目移到归档区 `think.archive.md`，不再进入加载链 | `sofagent/loop-check.md` | 修改 |

> 💡 **设计原则**：记忆系统的敌人不是容量，是噪声。宁可少记，不可记错。

## 五、scoring 判断力维度（P2）

> 评审原话：「scoring 奖励"多跑多成功"——但应该奖励"知道什么不该做、何时主动停"。任务准入拒绝一次应该比完成一个简单任务值更多分。当前评分体系和治理目标在打架。」

### 问题

| # | 评审发现 | 严重程度 |
|:--:|------|:--:|
| S1 | **scoring 维度没有「判断力」指标**：Agent 主动拒绝高风险任务（如「帮我把数据库删了」）是治理层最想要的正面行为，但当前评分体系把它算作「未完成任务」 | P2 |
| S2 | **LLM 自评自噬风险**：[LLM自评] 标记降权 ×0.5 不足以阻止偏差正反馈——Agent 的评分偏见仍可能被下一轮反思吸收 | P1 |

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| S1 | **scoring 新增「判断力」维度**：九维——任务准入拒绝计正向分（拒绝 1 次高风险任务 +0.5），不该做但做了计负向分。权重初始 0.1（样本不足），随数据积累调高 | `sofagent/loop-check.md` closure 模式 | 修改 |
| S2 | **LLM 自评降权 ×0.5 → ×0.3**：进一步降低自评权重。think.md 反思头部加警告注释：「以下反思均为 LLM 自评，可能存在确认偏误。」 | `sofagent/loop-check.md` | 修改 |

> 💡 **设计决策**：判断力不和生产力放在同一个总分里加权——分开计、分开看。「你很能跑」和「你很会判断什么不该跑」是两件事。

## 六、编排引擎加固（P2）

> 评审原话：「engine.md 有编排深度但无检查点定义——不是跑完才检查，是每步都有验证。失败分支是隐式兜底——"如果失败了就降级"不是失败处理，是投降。」

### 问题

| # | 评审发现 | 严重程度 |
|:--:|------|:--:|
| O1 | **编排流程无内嵌检查点**：每步完成后不验证就直接进下一步——违反铁律 #3「验证再继续」 | P2 |
| O2 | **失败分支是隐式兜底**：只有一个 catch-all 的「失败→降级」，没有按失败类型分类处理 | P2 |
| O3 | **缺少显式停止条件**：「执行到跑不动为止」不是编排——没有一个节点 Agent 会主动说「到此为止，再往下做质量会下降」 | P2 |

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| O1 | **编排流程每步加验证节点**：engine.md 编排流程中，每步完成后必须跑对应的验证（代码步→bash -n/test，文档步→拼写/链接检查，数据步→行数/格式验证）。验证失败 → 不进入下一步 | `sofagent/engine.md` | 修改 |
| O2 | **6 个显式失败分支**：测试失败（单步回溯）、连续两次失败（回滚 + 通知用户）、改动太大（超文件数/行数阈值→暂停）、任务冲突（多个子任务改同一文件→合并策略）、多 Agent 矛盾（子 Agent 结论冲突→主 Agent 裁决）、成本超预算（超 token 限额→降级） | `sofagent/engine.md` | 修改 |
| O3 | **task-aware 目标契约加停止条件字段**：契约模板新增「停止条件」字段——什么情况下 Agent 应主动停止执行并汇报，而非继续尝试 | `sofagent/task-aware.md` | 修改 |
| O4 | **task-orchestrate.sh 加 --max-retries**：编排脚本新增重试上限参数，默认 3 次。超限 → 汇报失败原因，不继续尝试 | `sofagent/scripts/task-orchestrate.sh` | 修改 |

## 七、rules.md → Agent 运行规范升级（P2）

> 来源：循环工程师笔记——Scale File 应从提示词模板升级为 Agent 运行规范。

### 问题

rules.md 当前定位是「自定义规则」——用户往里面写几条偏好。但作为一个长时间运行的 Agent 治理层，应该有一个完整的「运行规范」概念：项目目标 + 验收标准 + 风险边界 + 停止条件。

### 解决方案

| # | 改动 | 文件 | 类型 |
|:--:|------|------|:--:|
| R1 | **DEVELOPMENT.md 中 rules.md 描述升级**：从「自定义规则」改为「Agent 运行规范」——描述应包含项目目标、验收标准、风险边界、停止条件。功能不变，命名和描述升级 | `DEVELOPMENT.md` | 修改 |
| R2 | **HANDBOOK.md §二 rules.md 描述升级**：同上——从「你的规则」变为「你的运行规范」 | `HANDBOOK.md` | 修改 |

> 💡 **功能不变，概念升级**：不要求用户写更多东西——只是让文件定位从一个「偏好清单」变成一个「运行合同」。

---

1. **v0.73 只碰运行时逻辑，不碰门面**：loop-check + engine + entry-gate + task-aware + task-orchestrate + task-record + task-closure——这些都是 Agent 执行时的逻辑文件，不在会话展示层面
2. **任务闸用「流程钉」不是「技术拦截」**：prompt 层无法做技术拦截。让 Agent 自己声明「我通过了检查」——声明本身不是拦截，但让跳步从「悄悄跳过」变成「公然说谎」
3. **记忆规则宁可少记不可记错**：写入标准比写入容量重要 10 倍。一条错误记忆的破坏力大于十条正确记忆的防护力
4. **判断力和生产力分开计分**：不放在同一个总分里——一个是「你多能跑」，一个是「你多会判断什么不该跑」
5. **失败分支显式设计**：不靠 catch-all 的「失败→降级」。每条失败路径有名字、有触发条件、有处理策略
6. **rules.md 升级不改功能**：只是概念更清晰——「运行规范」比「自定义规则」更准确地描述了它的作用
7. **engine.md 与 v0.72 不冲突**：v0.72 改 engine.md 头部 A2 节（能力对比表），v0.73 改 engine.md 编排流程段（检查点 + 失败分支）。两者在文件中的物理位置不重叠。执行 v0.73 时先确认 v0.72 的 A2 表格已就位
8. **日志加载优化推到 v0.8**：_recent.md 的可靠性依赖闭环定时触发——没有 daemon，Agent 跳过 closure 就会产生过期数据。v0.8 daemon 有定时触发能力后再做
9. **constitution/ 扁平化做无感升级**：install.sh 检测旧路径自动迁移到新路径 + 删旧目录。老用户跑一次 install.sh 就升级完成，不需要手动搬文件。`~/.openclaw/rules.md` 用户全局规则不动——它是用户自定义空间，和 sofagent 部署的 rules.md 语义不同
