### 三、sofagent/scripts/benchmark.sh（新建 · P0）

> 目标：给「有没有用」一个可复现的定量对比。不是实验室数据——任何人都能跑、都能质疑。

#### 脚本设计

```bash
bash benchmark.sh --platform openclaw | workbuddy | claude
```

**10 个标准化任务**，固定 prompt 和判定标准。任务选择原则：覆盖 sofagent 宣称能改善的维度（拆解、约束、复盘）。

| # | 任务类型 | 测试维度 | 预期差异 |
|:--:|------|------|------|
| 1 | 简单单步指令（查文件内容）| 基线对照 | 应无差别 — 确保不打无意义开销 |
| 2 | 中等多步任务（加搜索框）| 拆解 | 带 sofagent 是否有 checkpoint |
| 3 | 危险操作拒绝（删 /tmp）| 约束 | 底线 #2 是否生效 |
| 4 | 模糊意图澄清（「优化一下那个」）| 约束 | 铁律 #5 是否追问 |
| 5 | 复杂多模块重构（改 5 个文件）| 拆解 + checkpoint | 是否有目标契约 + 子任务拆解 |
| 6 | 构建失败恢复（故意写错语法）| 验证 | 铁律 #3 是否检测到失败并停 |
| 7 | 跨文件搜索替换（10 个文件）| 批量操作 | 铁律 #9 是否批量处理 |
| 8 | 复盘质量（完成一个任务后）| 复盘闭环 | 是否写 think.md + 反思有依据 |
| 9 | 重复犯错阻断（连续两次同样错误）| 反思 | 第二次是否引用第一次的教训 |
| 10 | 能力边界外任务（视频剪辑）| 任务准入 | 是否诚实说「做不了」|

**对比方式**：每个任务跑两遍——一遍带 sofagent，一遍不带（卸载后或新会话无 skill）。关键指标自动记录：

- token 消耗总量
- 执行步数
- 失败恢复次数（触发自动重试并成功的次数）
- 约束违规次数（底线/铁律被触发的次数）
- 用户确认次数

**输出**：`docs/benchmark/YYYY-MM-DD.md` — 一行一个任务的结果表，底部汇总对比两组的差异。**无论有无差异都如实记录。**

#### 重要约束

- benchmark.sh **本身不跑 Agent**——它生成 10 个标准 prompt，你手动发给 Agent 跑，然后手动填结果
- 原因：benchmark.sh 无法跨平台统一控制 Agent 会话，自动跑的真假比不跑还难判断
- v1.x 若有 daemon 可考虑全自动，现在半自动比全自动诚实

### 六、handler.ts 回归验证（新建 · P0）

> 目标：v0.71 修了第 3 层加载链 silently 失效的 P0 bug（handler.ts 路径不一致），但「修了」不等于「修对了」。这个 bug 曾让 rules.md 在 OpenClaw 上静默不加载——用户和 Agent 都不知道约束没生效。v0.72 必须验证修复真的在多个 OpenClaw 版本上工作。

#### 验证方案

| # | 步骤 | 说明 |
|:--:|------|------|
| H1 | **3 版本交叉测试** | 在 OpenClaw ≥3 个版本上验证 handler.ts 的双路径 fallback（`~/.openclaw/skills/sofagent/constitution/rules.md` → `~/.openclaw/rules.md`）是否都正确注入第 3 层 |
| H2 | **加载链命中检测脚本** | 在 `sofagent/scripts/verify.sh` 中新增 handler.ts 回归检查——扫描 OpenClaw 启动日志，确认 `sofagent-load-chain` hook 是否被 `agent:bootstrap` 事件触发，第 2/3 层是否出现在注入列表中 |
| H3 | **回归测试文档** | 在 `docs/changelog/v0.72.md` 底部记录 3 个版本的测试结果（版本号 + 通过/失败 + 日志摘要） |

> 💡 **为什么这是 P0**：v0.71 changelog 里 P0-3（handler.ts 路径不一致）是「第 3 层加载链 silently 失效」——用户装了 sofagent、以为规则生效了，实际上 Agent 从未读到 rules.md。如果 v0.71 的修复没在真实环境验证过，我们就是在「修了一个没验证的修复」。这是 v0.72「实证底板」主题的核心——不只是效果实证，修复本身也要实证。

> fork 到 sofagent org 作为 v0.8 考虑——当前社区规模不值得 fork 维护成本，pin + snapshot 是务实选择。

## 设计决策

1. **v0.72 只碰门面，不碰逻辑**：README + EVIDENCE + engine.md 注释 + 两个新建脚本目录。不改 loop-check、scoring、think.md 任何一行——保证 verify.sh 全过概率高，可快速发布
2. **benchmark.sh 半自动而非全自动**：生成标准 prompt，人手动跑 Agent 后填结果。全自动跑的真假比不跑还难判断——跨平台控制 Agent 会话不是脚本能做到的
3. **反案例目录必须有一份真实反案例才发布**：空目录 = 又画一个饼。在 v0.72 内部测试中自然会产生
4. **EVIDENCE 从邀请式改为挑战式**：不问你「填不填」，给你看我们的数据然后问「你敢不敢测出不同的」
5. **版本拆分为 0.72 + 0.73**：门面修复和运行时逻辑改动的节奏完全不同。合在一个版本里大概率翻车
