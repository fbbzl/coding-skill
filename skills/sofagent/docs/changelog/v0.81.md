## 第一部分：daemon 核心骨架

### 要解决什么

加载链靠 Agent 自觉。跨 session 的经验（think.md）和规则（rules.md）能不能被读到，完全看 Agent 在下一个 session 是否愿意读。daemon 填的是「session 外触发器」这个空白。

v0.81 只做骨架——让进程存在、不崩、能监控。**不做注入逻辑**（记忆压缩路线已废弃，见「不包含」段）。

### 范围

| 做 | 不做 |
|---|------|
| daemon.sh + daemon-lib.sh（文件监控 + 进程检测 + JSON 读写） | ❌ Windows 不支持（见下方说明） |
| launchd（macOS）/ systemd（Linux）注册 | 定时任务调度（后续版本） |
| daemon-install.sh + daemon-uninstall.sh + daemon-status.sh | daemon 接管 cleanup（后续版本） |
| install.sh Step 6b / verify.sh / uninstall.sh 集成 | 记忆压缩 / reminder.md 注入（路线已废弃） |
| GitHub Actions CI（Linux systemd 测试） | |
| 五平台进程检测 + 能力矩阵验证 | |

#### 为什么 Windows 不支持

| 原因 | 说明 |
|------|------|
| **bash 非原生** | Windows 用 PowerShell / CMD，bash 需要额外装 Git Bash 或 WSL——违反「零外部依赖」 |
| **后台服务机制不同** | macOS 用 launchd，Linux 用 systemd，Windows 用「Windows 服务」API，三者完全不兼容 |
| **改动成本** | 支持 Windows 需要重写一套 daemon 注册逻辑，投入产出比不划算 |

**Windows 用户的体验**：装 sofagent 照样能用——宪法层（4 底线 + 10 铁律）正常加载，治理规则照样生效。只是 **daemon 后台监控那部分不生效**。这不是 bug，是架构边界——`install.sh --platform` 会检测系统，Windows 上自动跳过 daemon 安装步骤并提示用户。

## 第二部分：治理逻辑加固

### 加固 1：独立评判器（Maker-Checker）

> Loop Engineering 核心洞察：让执行任务的 Agent 评价自己的产出 = 自我肯定循环。银行转账——录入的人和复合的人必须是两个人。

**问题**：`loop-check.md` closure 模式写了 `session.spawn` 做工程隔离，但 `model: "flash"` 这行没说「必须和执行阶段不同模型」。如果执行用 Flash、验证也用 Flash——同一个模型看自己刚写的代码，它看到的不是结果，是「我为什么这么写」的自我说服过程。

**方案**：`loop-check.md` closure 模式「平台分级评审」段加两条规则：

**规则 A — 模型分离**：
```
闭环验证模型选择（优先级递减）：
1. 最优：与执行阶段不同的模型（执行 DeepSeek → 验证 GLM/Claude）
2. 可接受：同模型但不同 session（OpenClaw session.spawn 工程隔离）
3. 最低底线：同模型同 session + 重新 Read task/logs（非 OpenClaw 当前路径）
⛔ 禁止：同模型 + 凭执行记忆评审（零隔离）
```

**规则 B — 怀疑论提示**：
```
闭环验证的指令必须独立编写，不能复用执行阶段的指令。评判器提示应：
- 明确告知「这是别人执行的产出，你的任务是找问题」
- 要求逐项对照验收标准检查，不笼统问「有没有问题」
- 包含「列出至少 2 个潜在风险」类强制批判性要求
```
> 模型分离解决「视角污染」，怀疑论提示解决「指令污染」——即使换了模型，如果指令和执行时一样，评判器仍会顺着执行逻辑走。两条规则互补。

> 现实约束：作者当前只有 DeepSeek API。规则 A 写成分级递降——有条件时走最优，没条件时走可接受（session.spawn 工程隔离），不写死「必须不同模型」。规则 B 零成本，所有平台都必须做。

### 加固 2：防重复副作用（Idempotency）

> Agent 执行不可逆操作时，如果任务暂停又恢复、或子 Agent 重试，同一个操作可能被执行两次——发两封邮件、扣两次钱。

**问题**：`engine.md` 有 6 个失败分支和重试逻辑（`--max-retries 3`），但没有任何一个检查「这个操作上次是不是已经成功了」。重跑 = 可能重复执行副作用。

**方案**：`engine.md`「每步验证节点」表格之前，新增 idempotency pre-check 段落。覆盖 4 类不可逆操作：

| 操作类型 | 示例 | 检查方式 |
|---------|------|---------|
| **git push** | `git push origin main` | 查 task/logs 是否有同 branch + 同 commit-hash 的成功记录 |
| **rm -rf** | `rm -rf dist/` | 查目标路径是否已不存在（已删 = 已成功） |
| **外部 API**（POST/PUT/DELETE） | 发邮件、付款 | 查 task/logs 是否有同 operation-id 的成功响应 |
| **数据库写入** | INSERT / UPDATE | 查 task/logs 是否有同 row-key 的写入记录 |

**操作 ID 生成**：`echo "${task_id}${step_number}${resource}" | shasum -a 256 | cut -c1-16`

**流程**：
```
子 Agent 执行不可逆操作前：
1. 生成唯一操作 ID（task-id + step-number + resource-hash）
2. 查 task/logs 是否有同 ID 的成功记录
3. 有 → 跳过（标记「已执行，幂等跳过」）
4. 无 → 执行 → 写入 task/logs（ID + 执行状态 + 时间戳）
```

> 只覆盖 4 类不可逆操作。文件创建、代码修改等可重做操作不需要 idempotency 检查。

### 加固 3：「意图债」术语

> Loop Engineering 笔记术语——反复向 Agent 交代项目背景的成本。每次你告诉新 Agent「这个项目是什么、规则是什么、坑在哪」，都是在还一笔本可以固化的债。

**问题**：`ARCHITECTURE.md` §两层架构讲了地基（加载链）的作用，但没有一个词让用户秒懂「为什么需要 SKILL.md」。

**方案**：`ARCHITECTURE.md` §两层架构地基部分加一句话：

> SKILL.md 和 rules.md 还的是「意图债」——不用每次任务都重新交代项目背景、规则和已知坑点。

三个字，比任何长篇解释都有力。

### 加固 4：步数闸（Step Limiter）

> 来源：sofagent-dev 前身 `device/runtime/core/iteration-guard.js`（13,924 行重型编排引擎中少有的纯逻辑模块）。
> 前身用 JS 实现了 MAX_ITERATIONS=50 + GRACE_CALLS=3 的两段式预算控制——达到硬上限注入 BUDGET_EXHAUSTED 提示（让 Agent 自己收尾），超 MAX+GRACE 发射 TASK_FORCE_STOP（强制终止）。比纯 timeout 更精细。

**问题**：当前 `engine.md` 的循环控制只有 `--max-retries 3`（重试上限）和 timeout（硬截断），没有「一个 Session 内最多执行多少步」的预算概念。Agent 在无人值守的 daemon 场景下，可能在一个任务里反复调工具直到 timeout 才被 kill——浪费 token、拖慢链路。

**方案**：`engine.md` 新增步数闸段落。纯 bash 实现：

```bash
MAX_STEPS=50          # 硬上限
GRACE_STEPS=3         # 恩典步数（让 Agent 收尾输出）

# 每步 hook 递增计数器
step_count=$(cat "$STEP_COUNTER_FILE" 2>/dev/null || echo 0)
step_count=$((step_count + 1))
echo "$step_count" > "$STEP_COUNTER_FILE"

if [ "$step_count" -ge "$MAX_STEPS" ] && [ "$step_count" -lt "$((MAX_STEPS + GRACE_STEPS))" ]; then
    # 注入收尾提示：步数预算即将耗尽，请立即输出最终结果
    inject_budget_warning
elif [ "$step_count" -ge "$((MAX_STEPS + GRACE_STEPS))" ]; then
    # 强制终止
    force_stop "步数预算完全耗尽"
fi
```

> **为什么需要恩典期**：直接硬截断会丢失 Agent 的中间产出。恩典期给 Agent 3 步机会输出最终结果——比 timeout 暴力 kill 更优雅。前身 IterationGuard 的设计已经验证了这个思路。

### 加固 5：熔断闸（Circuit Breaker）

> 来源：sofagent-dev 前身 `device/runtime/core/behavior-validator.js` 的三态断路器模式。CLOSED → 连续失败 → OPEN（拒绝请求）→ 冷却 → HALF_OPEN（试探）→ 恢复/重新熔断。

**问题**：加固 2（防重复副作用）解决了「同一个操作被执行两次」，但没有解决「子 Agent 持续失败拖死整条链」。如果一个子 Agent 因为 API 限流、网络抖动、模型超载连续失败，当前 `engine.md` 的重试逻辑（`--max-retries 3`）会在每个调用点独立重试——如果问题持续存在，每个下游子 Agent 都会独立地撞墙 3 次后才放弃。N 个子 Agent × 3 次重试 = 3N 次无效调用。

**方案**：`engine.md` 新增熔断闸段落。维护 per-Agent 熔断状态文件，三态机：

```bash
CIRCUIT_FILE="$TASK_DIR/circuit_state"   # per-Agent
FAILURE_THRESHOLD=3                       # 连续失败 3 次 → 熔断
COOLDOWN_SECONDS=30                       # 熔断后冷却 30 秒

# 调用子 Agent 前检查熔断状态
state=$(get_circuit_state "$agent_id")
case "$state" in
    OPEN)
        if [ "$(time_since_opened "$agent_id")" -ge "$COOLDOWN_SECONDS" ]; then
            set_circuit_state "$agent_id" HALF_OPEN   # 冷却期满，试探
        else
            skip_agent "$agent_id" "熔断中，跳过"      # 冷却期未满，拒绝
            return 1
        fi
        ;;
esac

# 子 Agent 返回后更新状态
if agent_success; then
    set_circuit_state "$agent_id" CLOSED
    reset_failure_count "$agent_id"
else
    increment_failure_count "$agent_id"
    if [ "$(get_failure_count "$agent_id")" -ge "$FAILURE_THRESHOLD" ]; then
        set_circuit_state "$agent_id" OPEN
        log "Agent $agent_id 连续失败 ${FAILURE_THRESHOLD} 次，熔断"
    fi
fi
```

> **三态说明**：CLOSED（正常，放行）→ 连续失败达阈值 → OPEN（拒绝所有请求，冷却 30 秒）→ 冷却期满 → HALF_OPEN（放行 1 次试探）→ 成功回 CLOSED / 失败回 OPEN。防止子 Agent 雪崩。

> v0.81 是 sofagent 第一个有 daemon 的版本。daemon + 步数闸 + 熔断闸 + 幂等检查 + 评判器隔离——五项新能力，五个平台，全部需要验证。

### 各维度测试方法

#### 维度 1 · daemon 进程检测

daemon 的核心能力是「知道 Agent 是否在运行」。用 `pgrep` 检测各平台 Agent 进程：

| 平台 | 预期进程名 | 检测命令 | 命中判定 |
|------|------|------|------|
| OpenClaw | `openclaw` 或 `claude` | `pgrep -f "openclaw\|claude"` | 有 PID 输出 |
| WorkBuddy | `WorkBuddy` 或 `workbuddy` | `pgrep -f "[Ww]ork[Bb]uddy"` | 有 PID 输出 |
| Claude Code | `claude` 或 `Claude` | `pgrep -f "[Cc]laude"` | 有 PID 输出 |
| Codex | `codex` 或 `Codex` | `pgrep -f "[Cc]odex"` | 有 PID 输出 |
| Hermes | `hermes` 或 `Hermes` | `pgrep -f "[Hh]ermes"` | 有 PID 输出 |

**测试步骤**：
1. 启动目标平台的 Agent（打开一个会话）
2. 运行 `bash sofagent/scripts/daemon-status.sh --detect`
3. 记录 daemon.json 中 `detected_platforms` 字段
4. 关闭 Agent 会话
5. 再次运行，确认 `detected_platforms` 变空

**降级路径**：如果 pgrep 命不中（进程名不匹配），daemon 降级为「仅文件监控模式」——不检测 Agent 进程，只监控 think.md / rules.md 的 hash 变化。

#### 维度 2 · 步数闸生效

步数闸写进 `engine.md`，靠 Agent 读到后自觉执行。验证 Agent 是否在到达 MAX_STEPS 时停止调用工具：

**测试步骤**（每平台）：
1. 安装 sofagent，确认 engine.md 被加载
2. 给 Agent 一个需要 50+ 步的复杂任务（如「分析这个项目并写一份完整的架构报告」）
3. 观察 Agent 行为：
   - ✅ 生效：Agent 在 ~50 步时开始收尾输出最终结果
   - ❌ 不生效：Agent 无视步数限制，持续调工具直到平台 timeout
4. 记录 `task/logs/` 中的 step 计数

#### 维度 3 · 熔断闸生效

**测试步骤**（每平台）：
1. 安装 sofagent，确认 engine.md 被加载
2. 给 Agent 一个会触发子 Agent 连续失败的任务（如「调用一个不存在的 API 端点 5 次」）
3. 观察 Agent 行为：
   - ✅ 生效：连续失败 3 次后，Agent 主动报告熔断并跳过
   - ❌ 不生效：Agent 每次都独立重试 3 次，不做跨调用计数
4. 记录 `task/logs/` 中是否有 `circuit_open` 标记

#### 维度 4 · 幂等检查生效

**测试步骤**（每平台）：
1. 安装 sofagent，确认 engine.md 被加载
2. 让 Agent 执行一个 git push 操作
3. 中断任务，重新启动 Agent 执行相同的 push
4. 观察：
   - ✅ 生效：Agent 检查 task/logs 发现已成功，标记「幂等跳过」
   - ❌ 不生效：Agent 再次执行 push（git 会报 already up to date，但 Agent 没做 pre-check）

#### 维度 5 · 评判器隔离

**测试步骤**（每平台）：
1. 安装 sofagent，确认 loop-check.md 被加载
2. 让 Agent 完成一个任务并进入闭环验证
3. 观察闭环验证是否使用与执行不同的模型/session
4. 记录：
   - ✅ 最优：用了不同模型
   - ✅ 可接受：同模型不同 session
   - ⚠️ 底线：同模型同 session + 重新 Read
   - ❌ 违规：同模型 + 凭记忆评审

#### 维度 6~8 · 加载链命中率

延续 v0.6x 的加载链测试方法：

**测试步骤**（每平台 × 每层）：
1. 新建会话（清空上下文）
2. 让 Agent 执行一个简单任务（如「读一下 SKILL.md 告诉我 4 底线是什么」）
3. 检查 Agent 输出是否引用了对应文件的内容
4. 重复 3 次取命中率

## 关键决策记录

### 决策 1 · 纯 bash（不用 jq）

- **选择**：daemon.json 的 JSON 读写用纯 bash（grep + sed 解析扁平字段）
- **不选 jq**：jq 在 macOS 不预装（需 brew），违反 sofagent「零外部依赖」招牌
- **风险**：嵌套 JSON 操作不可用——但 daemon.json 全是扁平字段，不需要嵌套

### 决策 2 · daemon + 治理加固 + 平台验证三合一

- **v0.81**：daemon 核心骨架（进程能跑）+ 5 项治理加固 + 5 平台兼容性验证
- **后续版本**：待 v0.81 实测数据出来后再规划（记忆压缩路线已废弃）
- **理由**：先补漏洞 + 实测能力，才知道 daemon 到底在哪些平台上能用

### 决策 3 · 核心逻辑走 TDD

- daemon 核心函数（hash 比对、进程检测、JSON 读写、降级路径）走 TDD
- QA 严过关先写测试 → 工程师寇豆码 Red→Green→Refactor 实现
- 系统集成（launchd/systemd 注册、Agent 进程检测）走手动验证清单

### 决策 4 · Linux 测试走 GitHub Actions CI

- **选择**：ubuntu-latest 容器跑 systemd 测试
- **不选 Docker Desktop**：作者机器未安装 Docker，省磁盘 + 省内存
- **优势**：公开仓库免费无限分钟，push 自动验证，无需手动跑
- **限制**：launchd（macOS）无法在 CI 测——作者本地测

### 决策 5 · 治理加固与 daemon 合并到 v0.81

- **选择**：5 项治理加固（独立评判器 + 防重复副作用 + 意图债术语 + 步数闸 + 熔断闸）和 daemon 骨架一起做
- **理由**：先补治理逻辑漏洞，再给它加触发器。顺序不能反——给有漏洞的系统加触发器没意义

### 决策 6 · 独立评判器用分级规则（不写死「必须不同模型」）

- **现实约束**：作者当前只有 DeepSeek API
- **规则**：分级递降——最优（不同模型）→ 可接受（同模型不同 session）→ 最低底线（同模型 + 重新 Read）
- **铁律**：⛔ 禁止同模型 + 凭执行记忆评审

### 决策 7 · 步数闸和熔断闸从 sofagent-dev 前身代码提取

- **来源**：前身项目 `device/runtime/core/iteration-guard.js`（步数预算）和 `behavior-validator.js`（断路器）——13,924 行重型 JS 编排引擎中少有的纯逻辑模块
- **改名**：IterationGuard → **步数闸**，CircuitBreaker → **熔断闸**（与 SKILL.md「闸门」概念统一命名）
- **适配**：前身用 JS 实现，v0.81 用纯 bash 重写（符合 daemon 零外部依赖决策）
- **不搬的**：前身的 DAG TaskGraph、Receptionist 接待员引擎、Capacity 容量检测——这些是 13K 行重型编排的产物，与 Handbook「薄在复用」路线冲突
