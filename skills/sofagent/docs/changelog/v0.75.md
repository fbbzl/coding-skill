# v0.75 开发日志

> 2026-06-21 · 降低试用门槛 + 补可信度第一块数据
>
> v0.8（daemon）前的最后一个「不碰运行时」版本。

---

## 概述

v0.72 修门面，v0.73 修运行时，v0.74 修治理层自身。三个版本把「看起来像回事」做到了——文档诚实、运行时加固、易用性提升。

v0.74 发布后，GLM-5.2 做了一次双视角深度审查（GitHub 维护者 + 技术 VP/CTO），评分 7/10 和 5/10。审查结论：**设计精良的概念验证，正往「能放心推荐」走，但还没到。**

v0.75 要解决的，是审查里暴露的两个最大缺口：

1. **可信度缺口**：宣称「越用越聪明」，但 0 个第三方持续使用数据，也没有自己的 A/B 对比数据
2. **试用门槛缺口**：纯中文 README、无 demo gif、无架构图——GitHub 访客 30 秒内看不懂就跳走

v0.75 的定位：**不碰运行时逻辑，聚焦降低试用门槛 + 补可信度第一块数据。** 让看到这个项目的人更愿意试一下，让试过的人能看到数据。

---

## 问题来源

GLM-5.2 双视角深度审查（2026-06-21），全量通读 9 个核心文件：

| 视角 | 评分 | 判断 |
|------|:--:|------|
| GitHub 维护者（10K+ star 项目经验） | 7/10 | 诚实度杀手锏，文档碾压 90% 同类，但缺 demo/英文/持续数据 |
| 技术公司 CTO（50 人团队评估视角） | 5/10 | 概念验证阶段，不能上生产；明文/自评/单人维护三项致命风险 |

**审查的事实判断我们完全认可。** 以下是审查整理出的全部问题与候选方案，按优先级分级。

## 问题清单与候选方案

### 🔴 P0 — 会影响项目可信度 / 阻止采用

#### P0-A1 · EVIDENCE 零持续使用数据

> **审查原话**：「EVIDENCE.md 的 6 条全是一次性测试，0 条持续使用。核心宣称（越用越聪明、纪律提升）至今 0 个第三方长期数据支撑。」

| 维度 | 说明 |
|------|------|
| **问题** | `docs/EVIDENCE.md` 的「使用时长」列写满「一次性测试」，「持续使用 >1 周的用户数：待统计」。一个宣称「越用越聪明」的项目自己没有持续使用数据——信誉根基悬空 |
| **影响** | GitHub 维护者看到会想「你自己都没持续用，我怎么信」；CTO 看到会想「效果 0 证据，怎么向团队推荐」 |
| **根因** | 治理效果本身难以量化（「Agent 跑没跑偏」很难测）；项目处于早期，还没到 Beta 招募阶段 |
| **候选方案** | ① **作者自己先持续用 2 周**，填第一份「持续使用」数据（最真实，但样本=1）；② **跑 benchmark.sh 产出第一份公开 A/B 对比数据**（10 个标准化任务 × 带不带 sofagent），放进 EVIDENCE.md——这是当前能做的、最快的、可复现的数据来源 |
| **v0.75 做** | ✅ 方案②——跑 benchmark.sh，产出第一份 A/B 数据。哪怕结果是「有改善但不显著」，也比「效果？我们不知道」强 10 倍 |
| **推到后续** | 方案①作者持续用 2 周（贯穿 v0.75 开发期，数据自然积累）；Beta 招募推到 v0.9（需 daemon 先稳定） |

#### P0-A2 · 无 demo gif / screenshot

> **审查原话**：「README 全是文字 + ASCII 图，没有一个可视化展示。对 GitHub 转化率影响极大——搜索结果点进来的人，70% 在看文字前就跳走了。」

| 维度 | 说明 |
|------|------|
| **问题** | `images/` 目录只有 logo（1 个 png），`README.md` 无 demo gif、无架构图、无截图。纯文字 README 在 GitHub 的转化率极低 |
| **影响** | 访客 30 秒内看不懂就跳走；「Agent 自动拆解任务」用文字描述不如 15 秒 gif 直观 |
| **候选方案** | ① **demo.gif**：左边裸 Agent 跑偏 vs 右边 sofagent 约束后正常（15 秒内）；② **架构图**：两层架构（地基 + 引擎）+ 加载链流程的 SVG/PNG；③ **截图**：verify.sh 输出截图 + think.md 样例截图 |
| **v0.75 做** | ✅ 架构图用 Mermaid 写进 README（GitHub 原生渲染）。demo.gif 推到后续版本（需人工录屏，已委托同事录制，录好后在下一个版本补入） |

#### P0-B1 · 数据全明文，无加密（企业致命）

> **审查原话**：「`.sofagent/` 下 task/logs + think.md 含任务记录、代码片段、API 响应摘要，全部明文。企业内任何有文件系统访问权限的人都能读取所有 Agent 的任务历史。GDPR / 等保 / SOC2 场景直接出局。」

| 维度 | 说明 |
|------|------|
| **问题** | install.sh 做了 `chmod 700`，但那只是文件系统权限，不是加密。think.md 可能被 LLM 提炼反思时无意写入敏感信息 |
| **影响** | 企业生产环境直接否决；高安全场景（金融/医疗/政务）出局 |
| **候选方案** | **age 加密**（ROADMAP v0.9 已规划）——age 只要一个二进制文件，零配置，符合「零依赖」哲学。默认关闭，不影响老用户 |
| **v0.75 做** | ❌ 不做。这是 v0.9 企业级的核心交付物，v0.75 碰运行时逻辑会破坏 v0.74 的稳定性 |
| **v0.75 做什么** | 在 SECURITY.md 和 EVIDENCE.md 里**诚实强化风险声明**——让企业用户明确知道「现在不能用生产环境」 |

#### P0-B2 · LLM 自评无外部基准（企业致命）

> **审查原话**：「复盘评分由执行任务的 Agent 自己打分。企业需要的是『这个 Agent 靠不靠谱』的客观证据，现在是 Agent 自己说『我这次做得不错』。」

| 维度 | 说明 |
|------|------|
| **问题** | 非 OpenClaw 平台连工程隔离都没有，Agent 自评自批准。ARCHITECTURE 自己承认「Agent 可以提议修改，但不能自己批准」是理想，当前做不到 |
| **影响** | 可接受于探索性场景，不可用于生产决策 |
| **候选方案** | **外部评估器**（ROADMAP v1.x 已规划）——bash 脚本跑确定性代码评估，输出 JSON 分数，不依赖 LLM |
| **v0.75 做** | ❌ 不做。外部评估器是 v1.x 的交付物，需要先有 daemon + 持续数据基础 |
| **v0.75 做什么** | 同 P0-B1——诚实声明 + benchmark.sh A/B 数据提供**部分客观证据**（虽然不是外部评估器，但比纯自评强） |

#### P0-B3 · 单人维护，abandoned 风险（企业致命）

> **审查原话**：「如果孔放勋明天不维护了，谁能接？『一人项目』在很多公司的采购白名单里直接被刷掉。」

| 维度 | 说明 |
|------|------|
| **问题** | README / CONTRIBUTING 明确标注维护者一人，所有代码 AI 生成。企业采用治理工具，工具本身 abandoned 则迁移成本极高 |
| **影响** | 很多公司的采购白名单直接刷掉单人项目 |
| **候选方案** | ① **招募 Co-maintainer**——明确需要什么技能（bash 跨平台 / OpenClaw hook / 安全审计）；② **纯 MD 格式缓解 vendor lock-in**——数据可读、可迁移；③ **社区贡献者培养路径**——从 Issue → PR → co-maintainer 的阶梯 |
| **v0.75 做** | ✅ 方案①——README / CONTRIBUTING 加「Seeking Co-maintainers」段；方案②强化已有优势（文档说明数据格式可迁移） |
| **推到后续** | 实际招募到 co-maintainer 取决于社区增长，不是写文档能解决的 |

> **审查原话**：「纯中文 README 把 80% 的潜在 star 挡在门外。一个治理 AI Agent 的项目，目标用户是全球开发者。」

> **审查原话**：「ROADMAP『欢迎参与』列了 6 种贡献方式但偏轻量，没有『寻求 co-maintainer』的重度贡献招募。」

| 维度 | 说明 |
|------|------|
| **问题** | 现有贡献指南都是轻量贡献（翻译/FAQ/测试），想深度参与的人不知道从哪入手 |
| **候选方案** | CONTRIBUTING 加「Seeking Co-maintainers」段：明确需要什么技能、commit 权限怎么给、从 contributor 到 maintainer 的路径 |
| **v0.75 做** | ✅ 与 P0-B3 合并处理 |

#### P1-A3 · LICENSE 双声明不清晰

> **审查原话**：「README badge 写 MIT + CC-BY-4.0，但没解释分界。维护者看到混合 license 第一反应是『能商用吗？』」

| 维度 | 说明 |
|------|------|

| **影响** | 企业法务评估时困惑；影响商用采用 |

#### P1-A4 · 脚本版本号 pin 了但不一致

> **审查原话**：「install.sh pin agency-orchestrator@0.7.5，但 verify.sh 没检查 ao 版本下限。用户装了旧版 ao，verify 会 pass 但 compose 可能行为异常。」

| 维度 | 说明 |
|------|------|
| **问题** | install.sh L248 pin 了 ao 版本，verify.sh L430-440 只检查 `ao compose --version` 是否有输出，没检查是否 ≥ 要求版本 |

| **v0.75 做** | ✅ verify.sh ao 版本下限检查，约 10 行 bash |

## 版本定位

```
v0.72 门面治理 → v0.73 运行时加固 → v0.74 治理层自身治理 → v0.75 降低门槛 + 补数据
   ↓                 ↓                    ↓                      ↓
 让人不会被骗      让东西真的能用       让治理层自己也被治理     让更多人愿意试 + 看到效果数据
```

v0.75 是 v0.8（daemon）前的最后一个「不碰运行时」版本。目标：把审查暴露的门面和可信度缺口补到「值得 star + 愿意试一下」的程度，为 v0.8 daemon 的重磅改动蓄能。

### 决策 4 · Co-maintainer 招募——三级权限阶梯

> sofagent 是小项目，不设「申请制」——设门槛只会把人挡在外面。采用「贡献自然累积，作者主动邀请」模式。

**三级权限**：

| 级别 | 条件 | 能做什么 |
|------|------|---------|
| **Contributor**（任何人） | 无门槛 | 提 Issue / 发 PR |
| **Triage** | 合并 PR ≥1 个 **或** 有效 Issue ≥3 个 | 分流 Issue / 打标签 / 回复用户问题 / 关闭无效 Issue |
| **Co-maintainer** | 合并 PR ≥5 个 **+** 持续贡献 ≥2 个月 **+** 作者邀请 | review 和合并别人的 PR（不能直接 push main） |

**版本发布 / 架构决策**：目前只有作者。等 co-maintainer 稳定贡献 6 个月以上再谈。

**需要的技能**（写在招募段里，让有这些技能的人知道「这里需要我」）：
- bash BSD/macOS 兼容性（跨平台脚本是最大的工程债）
- OpenClaw hook 开发（TypeScript）
- 安全审计（企业级落地的前置条件）
- 英文文档（国际化最大瓶颈）

**为什么这么定**：
- 不设申请制 → 小项目设门槛只会挡人
- Triage 只需 1 个 PR → 降低「第一次参与」的心理门槛，让人先进来
- Co-maintainer 需 5 PR + 2 个月 → 保证对项目有足够了解，不会乱合并
- 不给 push main → 单人项目的主干保护，防止不可逆操作
- 技能需求公开 → 有这些技能的人能快速判断「我是不是你们要的人」

### 决策 5 · CI/CD 示例——只做 GitHub Actions

- **做 GitHub Actions**：覆盖面最广（sofagent 本身就托管在 GitHub），YAML 示例展示 verify.sh --json 怎么接入 CI
- **不做 GitLab CI / Jenkins**：维护成本高，等有 GitLab 用户提 Issue 再补。社区驱动
- **migration checklist**：独立文档段，列从「现有 Agent prompt」到「加 sofagent 约束层」的步骤

### 决策 6 · 发版节奏——一起发 v0.75

- **不分 v0.75.1 / v0.75.2**——12 项交付物体量不大，一次发版干净利落
- **执行顺序**：文档类改动（#3-#5 #7-#12）→ 脚本类改动（#6 #9 #10）→ 素材类（#2 录屏画图）→ benchmark 跑数据（#1）→ 全局版本号同步 → verify.sh 自检 → 发版
