# 代码库 RAG 规范

## 定位

代码库 RAG（Retrieval-Augmented Generation）用于为 AI 编码助手提供高质量、相关性强的代码上下文，避免 AI 基于幻觉或过时信息作答。

## 核心流程

```
代码库 → 解析 → 分块 → 向量化/结构化索引 → 检索 → 注入上下文 → AI 推理
```

## 索引范围

### 必须索引

- 源代码文件（按语言过滤）
- API 契约、接口定义
- 数据库 schema 和迁移脚本
- 配置文件（脱敏后）
- 测试文件
- 文档：README、API 文档、架构说明

### 必须排除

- 依赖目录：`node_modules/`、`target/`、`.venv/` 等
- 构建产物
- secrets、证书、私钥文件
- 日志文件
- 二进制文件
- `.git/`、`.idea/`、`.codegraph/`

## 分块策略

| 粒度 | 适用场景 | 示例 |
|---|---|---|
| 函数/方法 | 代码生成、bug 定位 | 一个方法作为一个 chunk |
| 类/模块 | 设计审查、依赖分析 | 一个类作为一个 chunk |
| 文件 | 文档、配置文件 | 一个文件作为一个 chunk |
| 调用链 | 跨文件分析 | 从入口函数展开的调用树 |

分块时应保留元数据：文件路径、行号、所属模块、调用关系、依赖关系。

## 元数据设计

每个 chunk 至少包含：

```yaml
chunk_id:
file_path:
start_line:
end_line:
module:
symbols: [函数名、类名、接口名]
callers: [调用方列表]
callees: [被调用方列表]
doc_refs: [关联文档路径]
last_modified:
```

## 检索策略

1. **混合检索**：向量相似度 + 关键词匹配 + 结构化元数据
2. **调用图扩展**：找到相关函数后，向上追溯调用方，向下展开被调用方
3. **最近修改优先**：对活跃文件提高权重
4. **问题驱动**：根据用户问题类型选择检索粒度

## 与 CodeGraph 结合

- CodeGraph 提供代码结构图谱（节点、边、调用关系）
- RAG 使用 CodeGraph 的关系数据增强检索
- 对复杂问题，先用 CodeGraph 定位相关子图，再提取代码片段

## 上下文注入原则

- 只注入与当前任务最相关的上下文，避免 prompt 过长
- 注入时保留文件路径和行号，便于 AI 引用
- 对多文件关联问题，先注入关系图摘要，再注入关键代码
- 定期清理过期索引，避免 AI 基于旧代码作答

## 质量门禁

- [ ] 索引范围明确，无敏感信息
- [ ] 分块粒度合理，保留足够元数据
- [ ] 检索结果与问题相关度高
- [ ] 注入上下文不超出模型窗口限制
- [ ] 索引更新机制完善（代码变更后自动/手动重建）

## 反模式

- 把整个代码库一次性塞给 AI
- 索引 secrets 或生产配置
- 分块过小导致丢失上下文，或过大导致噪声
- 只依赖向量检索，忽略调用关系和结构化元数据
- 索引长期不更新

## 与 opencode 联动

- dev 子代理在开发前使用 codebase-rag 检索相关代码
- cr 子代理在审查时检索受影响范围
- 与 `skills/ai-memory/SKILL.md` 联动：记忆中也记录代码库关键约定，减少重复检索
