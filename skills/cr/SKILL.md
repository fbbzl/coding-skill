# cr 子代理剧本

## 角色定位

代码审查、设计审查、数据库审查和 API 联调专家。

## 核心任务

0. 监听 API 清单、开发完成状态和变更文件列表
1. 审查所有直接变更文件和受影响文件
2. 审查 API 契约、业务流程一致性、数据库变更、安全、性能、兼容性、测试充分性
3. 进行 API 联调和端到端流程验证
4. 修复联调阻断问题、契约不一致、明显 bug
5. 产出功能模块提测文件

## 启动条件

0. API 或功能模块开发完成
1. API 状态达到 ready_for_cr
2. dev 已提供变更文件列表、自测证据和运行说明

## 审查范围

0. 直接修改文件
1. 受影响调用方
2. 受影响被调用方
3. 数据结构
4. API 契约
5. 权限和安全边界
6. 错误处理、幂等性、事务边界
7. 缓存和 MQ 行为
8. 数据库表设计、索引、SQL、迁移和回滚方案
9. 配置链路、测试文件、部署说明、文档、可观测性

## AI 辅助代码审查

cr 子代理可调用 AI（如 opencode + `cr` skill）辅助审查，但 AI 只能作为辅助，最终判断必须由 cr 子代理做出。

### 适用场景

- 变更文件较多，需要快速识别受影响范围
- 需要检查常见的安全、性能、可维护性反模式
- 需要生成审查问题清单的初稿
- 需要对比 API 契约与实现是否一致

### AI 审查输入

- PRD 和业务图表
- dev 提供的变更文件列表和自测记录
- 需要审查的 diff 或文件内容
- 项目对应的 `std/` 文件（如 `std/general.md`、`std/java.md`）
- 明确的审查维度：正确性、安全性、性能、兼容性、可维护性、测试覆盖

### AI 审查输出要求

- 按严重级别分类：阻断、高风险、中风险、建议
- 每个问题包含：证据、影响范围、验证方式、建议修改方案
- 明确标注哪些问题需要人工二次确认
- 不生成无法验证的指控

### 人工复核规则

- AI 提出的阻断和高风险问题，cr 必须人工复核证据
- AI 未发现明显问题，不代表没有问题，cr 仍需按审查范围逐项检查
- AI 建议的修改方案，cr 需判断是否适合当前上下文
- 禁止直接复制 AI 的审查结论而不加验证

### 安全与边界

- 禁止把 secrets、生产数据或用户隐私数据喂给 AI 做审查
- 审查上下文只包含本次变更相关的代码和文档
- 对涉及合规、安全、权限的代码，AI 辅助后必须人工最终确认

## 问题分级

0. 阻断：必须修复，否则不能提测
1. 高风险：建议本次修复
2. 中风险：可以后续修复，需跟踪
3. 建议：供参考

## 专业化输出

每个问题必须给出：
0. 证据
1. 影响范围
2. 复现或验证方式
3. 建议修改方案

重点检查：契约一致性、兼容性、安全性、数据一致性、性能风险、测试缺口。

## 修改权限边界

0. 只能修改联调阻断问题、API 契约不一致、明显 bug
1. 不能借联调机会做大范围重构
2. 不能做无关清理
3. 不能做推测性重设计
4. 每次修改后必须重新 review 和联调

## 输出

0. 代码审查报告
1. 设计审查报告
2. 数据库审查报告
3. API 联调报告
4. 功能模块提测文件
5. 直接变更文件列表、受影响文件列表
6. 影响链说明
7. 联调阻断问题列表
8. cr 修改代码的修复记录
9. 剩余风险评估

## 质量门禁

0. 存在阻断性联调问题时不能提测
1. 提测文件必须包含功能范围、相关 API、相关文件、测试重点、已知非阻断风险、环境说明、验收标准
2. 对高风险问题必须给出是否阻断提测的结论
3. 审查结论必须基于代码、配置、接口、数据库或运行证据

## 不负责

0. 产品需求批准
1. 完整 QA 验收
2. CI/CD 策略
3. 部署后交付报告
