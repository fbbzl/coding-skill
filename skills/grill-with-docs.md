# grill-with-docs

> **本项目禁止使用 `grill me`。统一使用 `grill-with-docs` 进行高风险问题澄清。**

`grill-with-docs` 是 `grill me` 的轻量替代方案。它不再进行持续多轮追问，而是通过阅读现有文档和方案，一次性提出最关键的几个澄清问题，快速达成可推进的共识。

## 适用场景

0. 需要澄清高风险问题：架构、验收标准、数据模型、权限、金额、合规、排期、交付范围
1. 已有 PRD、设计文档或方案草案
2. 希望减少来回对话次数

## 不适用场景

0. 完全没有文档，需要从零访谈
1. 用户明确要求深度讨论和持续追问

## 工作流程

### 第一步：阅读材料

0. 读取用户提供的 PRD / 设计文档 / 方案
1. 读取相关 `std/` 标准文件（如 api-design.md、database.md、security.md）
2. 读取相关 `skills/` 剧本（如 req.md、dev.md、openspec.md）
3. 如涉及代码变更，快速查看相关代码片段

### 第二步：识别高风险问题

从以下维度扫描文档，标记不清晰、有冲突或高风险的地方：

0. 架构：技术选型理由、扩展性、依赖关系
1. 验收标准：是否可测量、可验证
2. 数据模型：实体关系、字段类型、约束
3. 权限：角色、资源归属、越权风险
4. 金额/数值：精度、单位、计算逻辑
5. 合规：数据隐私、审计、保留策略
6. 排期：里程碑、依赖、阻塞点
7. 交付范围：包含什么、明确不包含什么

### 第三步：生成澄清问题

0. 最多提出 5 个核心问题
1. 每个问题说明：来源依据、风险点、默认假设（如果没有答案）
2. 问题格式：
   ```
   Q1: [具体问题]
   来源: [文档位置]
   风险: [不澄清的后果]
   默认假设: [如无回复则按此推进]
   ```

### 第四步：用户回答

0. 用户回答后，更新文档或记录结论
1. 未回答的问题保留为「假设」或「开放问题」
2. 当剩余问题不影响主流程推进时，停止追问

### 第五步：输出结论

0. 已确认的需求
1. 明确的假设
2. 仍开放的問題
3. 下一步行动

## 与 grill me 的区别

| 维度 | grill me | grill-with-docs |
|---|---|---|
| 驱动方式 | 持续对话访谈 | 文档驱动 |
| 提问方式 | 多轮逐步深入 | 一次性提出核心问题 |
| 停止条件 | 达成 shared understanding | 剩余不确定性可安全作为假设 |
| 适用场景 | 高风险、无文档 | 有文档、想快速推进 |
| 输出产物 | 对话记录 | 确认清单 + 假设清单 + 行动项 |

## 质量门禁

0. 问题必须基于文档证据，不凭空提出
1. 每个问题必须有默认假设
2. 问题数量不超过 5 个
3. 输出必须区分「已确认」「假设」「开放问题」
4. 不因为低风险不确定性阻塞主流程

## 使用示例

用户：请帮我 review 一下这个 PRD。

代理动作：
0. 阅读 PRD
1. 识别出 3 个高风险问题
2. 输出：
   ```
   已确认需求：...
   
   需要澄清的问题：
   Q1: 订单取消后优惠券是否退回？
      来源: PRD 第 3.2 节
      风险: 影响退款金额计算和库存回滚
      默认假设: 优惠券不退回
   
   Q2: 管理员能否取消他人订单？
      来源: PRD 第 4.1 节角色表
      风险: 权限设计越界
      默认假设: 仅超级管理员可操作
   
   Q3: 取消接口是否需要幂等？
      来源: std/api-design.md 幂等性章节
      风险: 重复请求导致多次退款
      默认假设: 需要幂等，通过 Idempotency-Key 实现
   ```
3. 根据用户回答更新结论，未回答的按默认假设推进
