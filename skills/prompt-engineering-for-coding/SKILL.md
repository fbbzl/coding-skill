# 编码场景 Prompt 工程

## 定位

本 skill 提供面向软件工程任务的 prompt 模板与反模式，帮助 dev、cr、qa 等子代理更高效地使用 opencode 或其他 AI 编码助手。

## 通用原则

1. **明确角色**：先定义 AI 扮演的角色，如"你是一名全栈开发专家"。
2. **明确目标**：说明要完成的任务，不要只给模糊指令。
3. **提供上下文**：给出相关文件路径、代码片段、需求背景、约束条件。
4. **指定输出格式**：要求结构化输出，如列表、表格、diff、JSON。
5. **分步骤执行**：复杂任务拆成多步，每步确认后再继续。
6. **限定范围**：明确哪些文件可以改，哪些不能改。

## 场景模板

### 1. 代码生成

```text
你是 [语言/框架] 专家。

目标：实现 [功能描述]。

上下文：
- 项目技术栈：[技术栈]
- 相关文件：[文件路径]
- 输入/输出：[数据格式]
- 约束：[性能/安全/兼容性要求]

要求：
1. 先输出实现方案，确认后再写代码
2. 代码遵循 [std/general.md, std/java.md 等]
3. 包含必要的异常处理和日志
4. 给出单元测试思路

输出格式：方案 + 代码 + 测试思路
```

### 2. 代码重构

```text
请审查以下代码，识别 [性能/可读性/可维护性/安全] 问题，并给出重构方案。

代码：
[代码片段]

要求：
1. 先列出问题，按严重级别分类
2. 给出重构后的代码
3. 说明每次修改的理由
4. 不改变外部行为
```

### 3. 代码解释

```text
请解释以下代码的功能、输入输出、关键逻辑和潜在风险。

代码：
[代码片段]

要求：
1. 用中文回答
2. 说明主要流程
3. 指出容易误解的地方
4. 列出可能的边界情况
```

### 4. Bug 定位

```text
以下代码出现 [现象描述]。

代码：
[代码片段]

错误信息：
[错误日志]

复现步骤：
1. ...
2. ...

要求：
1. 分析可能的原因
2. 给出最可能的根因
3. 提供修复代码
4. 说明验证方式
```

### 5. Code Review

```text
请对以下代码变更做审查。

变更文件：[文件路径]
Diff：
[diff 内容]

审查维度：
- 正确性
- 安全性
- 性能
- 可维护性
- 测试覆盖
- 是否符合 [std/xxx.md]

输出：
1. 按严重级别列出问题
2. 每个问题包含证据、影响、建议修改方案
3. 给出是否可通过审查的结论
```

### 6. 测试生成

```text
请为以下代码生成测试用例。

被测代码：
[代码片段]

要求：
1. 覆盖主路径、异常路径、边界条件
2. 使用 [测试框架]
3. 每个测试说明测试目的
4. 不修改被测代码
```

## 反模式

| 反模式 | 错误示例 | 改进方式 |
|---|---|---|
| 指令模糊 | "帮我优化这段代码" | "帮我优化这段代码的性能，目标是将时间复杂度降到 O(n)，不要改变接口" |
| 上下文缺失 | 只贴代码 | 补充需求背景、输入输出、约束 |
| 一次性要求太多 | "帮我设计、实现、测试、部署" | 分阶段，先设计再实现 |
| 不限制范围 | "随便改" | 明确可改文件和不可改文件 |
| 不验证输出 | 直接接受 AI 代码 | 要求 AI 说明理由，人工 review 后运行 |

## 与 opencode 结合

- 在 opencode 中，优先使用 `@file` 引用具体文件
- 对复杂任务，先用 prompt 让 opencode 输出方案，确认后再执行
- 把常用 prompt 模板保存到 `.ai-memory/prompt-templates.md`
- 定期回顾 prompt 效果，迭代优化

## 质量门禁

- [ ] Prompt 包含角色、目标、上下文、输出格式
- [ ] 复杂任务已拆分步骤
- [ ] 已限定修改范围
- [ ] AI 输出经过人工验证
